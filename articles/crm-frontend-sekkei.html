CRMフロントエンド設計

最終更新: 2026-03-18
Per Heim CRM v3.2のフロントエンド設計を定義する。ダッシュボード・カンバン・顧客カードの主要3画面を中心に、ディレクトリ構造、ルーティング、コンポーネント設計、CSS方針を包括的に記述する。

Per Heim CRM v3.2 のフロントエンドは、既存のAIヒアリング画面とは完全に独立したデスクトップ最適化レイアウトとして設計される。 既存の frontend/src/ 配下に crm/ ディレクトリを新設し、CrmLayout コンポーネントによる独自のサイドバー＋コンテンツ構造を持つ。 画面構成はダッシュボード（KPI・ファネル・タスク一覧）、カンバンビュー（フェーズ別ドラッグ&ドロップ）、顧客カード（AIプロファイル・タイムライン）の3主要画面を核とし、 タスク管理・反響受信トレイ・分析・設定画面を加えた計8画面で構成される。 詳細な統合方針はPer Heim CRM v3.2 統合設計書を、 バックエンドAPIはCRM API設計を、 データモデルはCRMデータモデル設計を参照のこと。

フロントエンドディレクトリ構造

CRMフロントエンドのコードは frontend/src/crm/ 配下に集約される。既存の pages/SalesConsole.tsx や pages/CompanyConsole.tsx には一切手を加えない。

4層構成の設計思想:

  • pages/ — ルーティングに対応する画面単位のコンポーネント。各ページは hooks を通じてデータを取得し、components を組み合わせてUIを構築する
  • components/ — 再利用可能なUIパーツ。カンバンカラム、タイムラインイベント、フェーズバッジなど、複数ページから参照される
  • hooks/ — CRM APIとの通信ロジックを集約。useCrmApi.ts が共通のfetch/エラーハンドリングを提供し、各ドメインhooksがラップする
  • types.ts — CustomerPhase, Customer, CrmTask 等のTypeScript型定義。データモデル設計と1:1対応

ルーティング設計

CRM画面のルーティングは React Router のネストルート構成を採用し、既存の RequireTenantAuth コンポーネントで認証ガードする。

// App.tsx に追加するルート
<Route element={<RequireTenantAuth />}>
  <Route path="crm" element={<CrmApp />}>
    <Route index element={<Dashboard />} />
    <Route path="customers" element={<CustomerList />} />
    <Route path="customers/:customerId" element={<CustomerDetail />} />
    <Route path="kanban" element={<CustomerKanban />} />
    <Route path="tasks" element={<TaskList />} />
    <Route path="inbox" element={<InquiryInbox />} />
    <Route path="analytics" element={<Analytics />} />
    <Route path="settings" element={<Settings />} />
  </Route>
</Route>

設計ポイント:

  • /crm がCRM機能全体のルートパスとなり、CrmApp コンポーネントが CrmLayout（サイドバー＋コンテンツ領域）を提供する
  • /crm にアクセスすると index ルートによりダッシュボードが表示される
  • RequireTenantAuth はテナントセッションCookieの有無を検証する既存コンポーネント。CRM固有の業務権限（admin/manager/sales）は各ページコンポーネント内で tenant_employees.role を参照して制御する
  • 顧客詳細は /crm/customers/:customerId でパラメータルーティング。カンバンの顧客カードクリック時もこのパスへ遷移する

CrmLayout の責務:

  • 左側に固定幅（--crm-sidebar-width: 220px）の CrmSidebar を配置
  • 右側のコンテンツ領域に React Router の <Outlet /> を配置し、選択されたページコンポーネントを描画
  • サイドバー最下部に「📎 招待管理」リンクを設置し、既存の SalesConsole 画面へスムーズに遷移可能にする
  • モバイル幅（768px以下）ではサイドバーをハンバーガーメニュー化し、コンテンツエリアをフルワイドにする

画面遷移の基本フロー:

  1. ログイン後、/crm にアクセスするとダッシュボードが表示される
  2. サイドバーから各画面へ遷移。URL直接入力によるディープリンクにも対応
  3. カンバンビューの顧客カードクリック → /crm/customers/:id へ遷移（顧客カード表示）
  4. 顧客カードの「戻る」ボタン → ブラウザ履歴の history.back() で直前画面に復帰

ダッシュボード

/crm のインデックスルートで表示されるダッシュボード画面は、営業チームの日常業務の起点となる画面である。4つのセクションで構成される。

KPIカード行

画面上部に横並びで5つのKPIカードを配置する:

カード表示値説明
新規反響件数直近7日間の新規反響数
ヒアリング中件数hearing_pending + hearing_active のフェーズにいる顧客数
来店予約件数visit_reserved フェーズの顧客数
成約件数当月の contracted フェーズ遷移数
完了率%ヒアリング完了率（hearing_done / hearing_active+hearing_pending+hearing_done）

デュアルファネル

左カラムにデュアルファネル（FunnelChart コンポーネント）を配置。反響→ヒアリング→来店→成約の各フェーズ別顧客数を棒グラフ形式で表示する。分析APIの GET /v1/tenant/crm/analytics/funnel からデータを取得する。

本日のタスク

右カラムに本日のタスクリストを配置。TaskItem コンポーネントによるチェックリスト形式で、期限が今日以前の未完了タスクを優先度順に表示する。チェックボックスクリックで即時完了処理を実行する。

直近の反響

画面下部に直近の反響リストを表示。ソース（SUUMO/HOME'S等）のアイコン、顧客名、連絡先（マスク表示）、経過時間を1行ずつ表示する。クリックで対応する顧客カードへ遷移する。

カンバンビュー

/crm/kanban で表示されるカンバンビューは、CustomerPhase（フェーズ）をカラムとして顧客カードを視覚的に管理する画面である。実装の詳細はカンバン実装詳細を参照。

フェーズ別カラム構成

KanbanBoard コンポーネントが横スクロール可能なカラムコンテナを提供し、各カラムは KanbanColumn コンポーネントで描画される。デフォルトの表示カラム:

カラムフェーズ値表示例
反響inquirySUUMO反響 田中様 5分前
ヒアリング待ちhearing_pendingURL発行済み 山田様
ヒアリング済みhearing_done完了率100% 佐藤様
来店予約visit_reserved3/20 14:00 鈴木様
成約contracted完了 高橋様

各カラムのヘッダーにはフェーズ名と件数を表示。カラム幅は --crm-kanban-column-width: 280px で統一する。

ドラッグ&ドロップでフェーズ遷移

顧客カードを別カラムにドラッグ&ドロップすることで、カンバンAPIの POST /v1/tenant/crm/kanban/move を呼び出してフェーズ遷移を実行する。遷移時の挙動:

  • ドロップ直後に楽観的UI更新（即座にカードを移動先カラムに表示）
  • API応答が成功なら確定、失敗なら元カラムにロールバック
  • フェーズ遷移ルールに違反する移動（例: inquiry → contracted への直接遷移）はAPIが拒否し、エラートースト表示
  • 遷移時にオプションで理由（reason）を入力できるモーダルを表示可能

顧客カード

/crm/customers/:customerId で表示される顧客カード画面は、1顧客のすべての情報を集約した詳細画面である。左右2ペインのレイアウトで構成される。

基本情報パネル（左ペイン）

左側の狭幅パネルに以下を縦積みで表示:

  • 基本情報: 顧客名、電話番号（マスク表示/クリックで表示）、メール、希望エリア
  • タスク: この顧客に紐づく未完了タスクの一覧（TaskItem）
  • アクションパネル: 「📞電話」「✉メール」「💬LINE」「📎URL発行」の4ボタン。各ボタンは追客操作を開始する

AIプロファイル表示（NeedsTags）

右側のメインペインのタブの一つとして、AIヒアリング結果から構造化された NeedsTags を表示する。データモデルの needs_tags フィールドに対応し、以下の3カテゴリで色分け表示する:

  • 🔴 dealbreaker（絶対NG条件）: 1階不可、築30年以上不可 など
  • 🟢 must（必須条件）: 駅徒歩10分以内、2LDK以上 など
  • 🟡 nice_to_have（あれば嬉しい条件）: オートロック、宅配ボックス など

NeedsProfile コンポーネントがこの表示を担当する。ヒアリング未完了の顧客には「AIプロファイル未取得」のプレースホルダーを表示する。

タイムライン

右側メインペインの「概要」タブにタイムラインを表示。Timeline + TimelineEvent コンポーネントで、顧客に紐づくすべてのイベントを時系列逆順で描画する:

  • 反響受信、ヒアリングURL送付、ヒアリング完了、フェーズ遷移、通話記録、メール送信、LINE送受信、来店記録、メモ追加 など
  • 各イベントはアイコン＋日時＋概要の1行で表示。クリックで詳細展開

リアルタイムデータ同期方式（SSE）

レビュー指摘 H11 対応: Firestore onSnapshot が Cookie 認証と非互換。MVP では SSE（Server-Sent Events）を推奨。Firestore Security Rules 不要で、認証は Cloud Run で完結する。

CRMフロントエンドにおけるリアルタイムデータ同期には、Firestore の onSnapshot ではなく SSE（Server-Sent Events）を採用する。

onSnapshot 非互換の背景

Per Heim CRM は tenant_session_cookie によるCookie ベース認証を採用している。Firestore クライアント SDK の onSnapshot は Firebase Auth トークンを前提としており、Cookie 認証との互換性がない。具体的には:

  • onSnapshot は Firestore Security Rules によるアクセス制御を前提とするが、CRM のテナント分離は CrmRepository のサーバー側ロジックで実装されている
  • Cookie 認証のセッション情報を Firestore Security Rules に渡す標準的な方法が存在しない
  • Firebase Auth を併用すると、認証基盤が二重化され、保守コストとセキュリティリスクが増大する

SSE アーキテクチャ

MVP では Cloud Run 上の SSE エンドポイントを通じてリアルタイム更新をフロントエンドにプッシュする:

  • エンドポイント: GET /v1/tenant/crm/events/stream
  • 認証: 既存の tenant_session_cookie で認証。Cloud Run のミドルウェアで CrmActor を解決し、データスコープに応じたイベントのみを配信する
  • イベント種別: customer_updated、task_created、inquiry_received、phase_changed
  • 再接続: EventSource API の自動再接続を活用。Last-Event-ID ヘッダーでイベントの再送を実現する
  • Firestore Security Rules: SSE 方式では不要。認証・認可は Cloud Run のサーバー側で完結する
// フロントエンド側の SSE 接続例（useCrmEvents.ts）
const useCrmEvents = (onEvent: (event: CrmEvent) => void) => {
  useEffect(() => {
    const source = new EventSource('/v1/tenant/crm/events/stream', {
      withCredentials: true  // Cookie 認証を送信
    });

    source.addEventListener('customer_updated', (e) => {
      onEvent(JSON.parse(e.data));
    });

    source.addEventListener('phase_changed', (e) => {
      onEvent(JSON.parse(e.data));
    });

    return () => source.close();
  }, [onEvent]);
};

レスポンシブ設計とモバイル対応

レビュー指摘 M10 対応: CRM画面はデスクトップ最適化を主軸とするが、タブレット・モバイルでの最低限の操作性を確保するためレスポンシブ設計の余地を組み込む。

ブレークポイント設計

CrmLayout にはレスポンシブ対応の余地を確保し、以下の3段階のブレークポイントでレイアウトを切り替える:

デバイス幅レイアウト変更
Desktop≥1024pxサイドバー固定220px＋コンテンツエリア。フル機能表示
Tablet768px〜1023pxサイドバーを折りたたみ可能（アイコンのみ表示 60px）。コンテンツエリア拡張
Mobile<768pxサイドバーをハンバーガーメニュー化。コンテンツエリアをフルワイド表示

カンバンビューのモバイル対応

カンバンビューはデスクトップでは横スクロール可能なカラムレイアウトを維持するが、モバイル（<768px）では以下の2モードを切替可能とする:

  • 横スクロールモード: 各カラムを 85vw 幅でスナップスクロール（scroll-snap-type: x mandatory）。スワイプでフェーズ間を移動
  • リスト表示モード: フェーズをアコーディオンヘッダーとし、縦方向に全カードを展開表示。KanbanBoard コンポーネントの viewMode prop で切替

デフォルトはリスト表示モードとし、ユーザーが任意にトグルできるUIを KanbanBoard のヘッダーに配置する。

タッチ操作対応

タッチデバイスではドラッグ&ドロップの操作性が低下するため、以下の対応を行う:

  • 長押しで D&D 発動: タッチデバイスでは 300ms の長押しでドラッグモードを開始。誤操作防止のため、通常タップとの区別を明確にする
  • 触覚フィードバック: navigator.vibrate(50) でドラッグ開始時に振動フィードバックを提供（対応デバイスのみ）
  • 代替操作: モバイルではカード上のコンテキストメニュー（「…」ボタン）からフェーズ遷移を選択する方式も提供。D&D に不慣れなユーザーへのフォールバック

アニメーション方針

レビュー指摘 L6 対応: CRM画面のアニメーション方針を定義する。カンバン D&D: @dnd-kit の内蔵アニメーション（CSS transitions）を使用。ページ遷移: React Router の route transitions（CSS-based）。通知トースト: CSS keyframes アニメーション。framer-motion はバンドルサイズの観点から MVP では不採用とし、Phase 4+ で必要に応じて検討する。prefers-reduced-motion メディアクエリを尊重すること。

共有モジュール境界の定義

レビュー指摘 M11 対応: design-system / hooks / auth の共有境界が未定義であったため、CRM固有モジュールと共有モジュールの責務境界を明確化する。

共有ディレクトリ構成

共有コンポーネントは frontend/src/shared/ ディレクトリに集約し、CRM固有のコードは frontend/src/crm/ に閉じる:

依存関係の方向ルール

モジュール間の import 方向を以下のルールで厳密に制御する。違反は eslint-plugin-import の no-restricted-paths ルールで CI で検出する:

  • shared → crm: OK — CRM は共有モジュールを自由に利用可能
  • shared → pages: OK — 既存ページも共有モジュールを利用可能
  • crm → pages: NG — CRM から既存ページへの import を禁止。既存画面との結合を防止
  • pages → crm: NG — 既存ページから CRM への import を禁止。逆方向の依存を排除
  • crm → shared: OK（上記 shared → crm と同義）
  • pages → shared: OK（上記 shared → pages と同義）
// .eslintrc.js — import 方向の強制
{
  rules: {
    'import/no-restricted-paths': ['error', {
      zones: [
        // crm → pages 禁止
        { target: './src/crm/', from: './src/pages/', message: 'CRM から既存 pages への import は禁止です' },
        // pages → crm 禁止
        { target: './src/pages/', from: './src/crm/', message: '既存 pages から CRM への import は禁止です' },
      ]
    }]
  }
}

共有モジュールのバージョニング方針

Per Heim プロジェクトは monorepo 構成のため、共有モジュールのバージョニングには以下の方針を採用する:

  • monorepo 内 workspace 方式: shared/ を内部 workspace パッケージ（@perhaim/shared）として定義し、package.json の workspaces で管理
  • セマンティックバージョニング: 共有モジュールの破壊的変更時は major バージョンを上げる。CRM・既存ページそれぞれが依存バージョンを明示
  • MVP 段階: monorepo 内の直接参照（相対パス import）で開始し、共有モジュールの安定後に workspace パッケージ化を検討する
  • 変更影響の可視化: 共有モジュールの変更時は、crm/ と pages/ の両方でテストを実行し、影響範囲を確認する CI ジョブを設定
