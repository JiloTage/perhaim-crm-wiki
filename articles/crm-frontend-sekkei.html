CRMフロントエンド設計

最終更新: 2026-03-18
Per Heim CRM v3.2のフロントエンド設計を定義する。ダッシュボード・カンバン・顧客カードの主要3画面を中心に、ディレクトリ構造、ルーティング、コンポーネント設計、CSS方針を包括的に記述する。

Per Heim CRM v3.2 のフロントエンドは、既存のAIヒアリング画面とは完全に独立したデスクトップ最適化レイアウトとして設計される。 既存の frontend/src/ 配下に crm/ ディレクトリを新設し、CrmLayout コンポーネントによる独自のサイドバー＋コンテンツ構造を持つ。 画面構成はダッシュボード（KPI・ファネル・タスク一覧）、カンバンビュー（フェーズ別ドラッグ&ドロップ）、顧客カード（AIプロファイル・タイムライン）の3主要画面を核とし、 タスク管理・反響受信トレイ・分析・設定画面を加えた計8画面で構成される。 詳細な統合方針はPer Heim CRM v3.2 統合設計書を、 バックエンドAPIはCRM API設計を、 データモデルはCRMデータモデル設計を参照のこと。

フロントエンドディレクトリ構造

CRMフロントエンドのコードは frontend/src/crm/ 配下に集約される。既存の pages/SalesConsole.tsx や pages/CompanyConsole.tsx には一切手を加えない。

frontend/src/
├── pages/                    # 既存ページ（変更しない）
│   ├── SalesConsole.tsx      # 既存：招待管理画面（維持）
│   └── CompanyConsole.tsx    # 既存：会社管理（維持）
│
├── crm/                      # ★新規：CRM画面
│   ├── CrmApp.tsx            # CRMルートコンポーネント（CrmLayout含む）
│   ├── pages/
│   │   ├── Dashboard.tsx        # ダッシュボード（KPI + ファネル）
│   │   ├── CustomerKanban.tsx   # カンバンビュー
│   │   ├── CustomerList.tsx     # 顧客一覧（テーブル）
│   │   ├── CustomerDetail.tsx   # 顧客カード（詳細）
│   │   ├── TaskList.tsx         # 本日のタスクリスト
│   │   ├── Analytics.tsx        # 分析ダッシュボード
│   │   ├── InquiryInbox.tsx     # 反響受信トレイ
│   │   └── Settings.tsx         # CRM設定
│   ├── components/

│   │   ├── KanbanColumn.tsx     # カンバンカラム
│   │   ├── CustomerCard.tsx     # 顧客カードコンポーネント

│   │   ├── TimelineEvent.tsx    # タイムラインイベント
│   │   ├── TaskItem.tsx         # タスク行
│   │   ├── PhaseTag.tsx         # フェーズバッジ
│   │   ├── NeedsProfile.tsx     # AIプロファイル表示（NeedsTags）
│   │   ├── CommunicationPanel.tsx # 追客パネル
│   │   ├── InquiryParser.tsx    # 反響パース結果表示
│   │   ├── FunnelChart.tsx      # ファネルチャート
│   │   └── CrmSidebar.tsx       # CRMサイドバー
│   ├── hooks/

│   │   ├── useCustomers.ts      # 顧客データ管理
│   │   ├── useTasks.ts          # タスク管理
│   │   └── useAnalytics.ts      # 分析データ
│   └── types.ts                 # CRM型定義（CustomerPhase等）
│
└── App.tsx                   # ルーティング追加

4層構成の設計思想:

  • pages/ — ルーティングに対応する画面単位のコンポーネント。各ページは hooks を通じてデータを取得し、components を組み合わせてUIを構築する
  • components/ — 再利用可能なUIパーツ。カンバンカラム、タイムラインイベント、フェーズバッジなど、複数ページから参照される
  • hooks/ — CRM APIとの通信ロジックを集約。useCrmApi.ts が共通のfetch/エラーハンドリングを提供し、各ドメインhooksがラップする
  • types.ts — CustomerPhase, Customer, CrmTask 等のTypeScript型定義。データモデル設計と1:1対応

ルーティング設計

CRM画面のルーティングは React Router のネストルート構成を採用し、既存の RequireTenantAuth コンポーネントで認証ガードする。

// App.tsx に追加するルート
<Route element={<RequireTenantAuth />}>
  <Route path="crm" element={<CrmApp />}>
    <Route index element={<Dashboard />} />
    <Route path="customers" element={<CustomerList />} />
    <Route path="customers/:customerId" element={<CustomerDetail />} />
    <Route path="kanban" element={<CustomerKanban />} />
    <Route path="tasks" element={<TaskList />} />
    <Route path="inbox" element={<InquiryInbox />} />
    <Route path="analytics" element={<Analytics />} />
    <Route path="settings" element={<Settings />} />
  </Route>
</Route>

設計ポイント:

  • /crm がCRM機能全体のルートパスとなり、CrmApp コンポーネントが CrmLayout（サイドバー＋コンテンツ領域）を提供する
  • /crm にアクセスすると index ルートによりダッシュボードが表示される
  • RequireTenantAuth はテナントセッションCookieの有無を検証する既存コンポーネント。CRM固有の業務権限（admin/manager/sales）は各ページコンポーネント内で tenant_employees.role を参照して制御する
  • 顧客詳細は /crm/customers/:customerId でパラメータルーティング。カンバンの顧客カードクリック時もこのパスへ遷移する

CrmLayout の責務:

  • 左側に固定幅（--crm-sidebar-width: 220px）の CrmSidebar を配置
  • 右側のコンテンツ領域に React Router の <Outlet /> を配置し、選択されたページコンポーネントを描画
  • サイドバー最下部に「📎 招待管理」リンクを設置し、既存の SalesConsole 画面へスムーズに遷移可能にする
  • モバイル幅（768px以下）ではサイドバーをハンバーガーメニュー化し、コンテンツエリアをフルワイドにする

画面遷移の基本フロー:

  1. ログイン後、/crm にアクセスするとダッシュボードが表示される
  2. サイドバーから各画面へ遷移。URL直接入力によるディープリンクにも対応
  3. カンバンビューの顧客カードクリック → /crm/customers/:id へ遷移（顧客カード表示）
  4. 顧客カードの「戻る」ボタン → ブラウザ履歴の history.back() で直前画面に復帰

ダッシュボード

/crm のインデックスルートで表示されるダッシュボード画面は、営業チームの日常業務の起点となる画面である。4つのセクションで構成される。

KPIカード行

画面上部に横並びで5つのKPIカードを配置する:

カード表示値説明
新規反響件数直近7日間の新規反響数
ヒアリング中件数hearing_pending + hearing_active のフェーズにいる顧客数
来店予約件数visit_reserved フェーズの顧客数
成約件数当月の contracted フェーズ遷移数
完了率%ヒアリング完了率（hearing_done / hearing_active+hearing_pending+hearing_done）

デュアルファネル

左カラムにデュアルファネル（FunnelChart コンポーネント）を配置。反響→ヒアリング→来店→成約の各フェーズ別顧客数を棒グラフ形式で表示する。分析APIの GET /v1/tenant/crm/analytics/funnel からデータを取得する。

本日のタスク

右カラムに本日のタスクリストを配置。TaskItem コンポーネントによるチェックリスト形式で、期限が今日以前の未完了タスクを優先度順に表示する。チェックボックスクリックで即時完了処理を実行する。

直近の反響

画面下部に直近の反響リストを表示。ソース（SUUMO/HOME'S等）のアイコン、顧客名、連絡先（マスク表示）、経過時間を1行ずつ表示する。クリックで対応する顧客カードへ遷移する。

カンバンビュー

/crm/kanban で表示されるカンバンビューは、CustomerPhase（フェーズ）をカラムとして顧客カードを視覚的に管理する画面である。実装の詳細はカンバン実装詳細を参照。

フェーズ別カラム構成

KanbanBoard コンポーネントが横スクロール可能なカラムコンテナを提供し、各カラムは KanbanColumn コンポーネントで描画される。デフォルトの表示カラム:

カラムフェーズ値表示例
反響inquirySUUMO反響 田中様 5分前
ヒアリング待ちhearing_pendingURL発行済み 山田様
ヒアリング済みhearing_done完了率100% 佐藤様
来店予約visit_reserved3/20 14:00 鈴木様
成約contracted完了 高橋様

各カラムのヘッダーにはフェーズ名と件数を表示。カラム幅は --crm-kanban-column-width: 280px で統一する。

ドラッグ&ドロップでフェーズ遷移

顧客カードを別カラムにドラッグ&ドロップすることで、カンバンAPIの POST /v1/tenant/crm/kanban/move を呼び出してフェーズ遷移を実行する。遷移時の挙動:

  • ドロップ直後に楽観的UI更新（即座にカードを移動先カラムに表示）
  • API応答が成功なら確定、失敗なら元カラムにロールバック
  • フェーズ遷移ルールに違反する移動（例: inquiry → contracted への直接遷移）はAPIが拒否し、エラートースト表示
  • 遷移時にオプションで理由（reason）を入力できるモーダルを表示可能

顧客カード

/crm/customers/:customerId で表示される顧客カード画面は、1顧客のすべての情報を集約した詳細画面である。左右2ペインのレイアウトで構成される。

基本情報パネル（左ペイン）

左側の狭幅パネルに以下を縦積みで表示:

  • 基本情報: 顧客名、電話番号（マスク表示/クリックで表示）、メール、希望エリア
  • タスク: この顧客に紐づく未完了タスクの一覧（TaskItem）
  • アクションパネル: 「📞電話」「✉メール」「💬LINE」「📎URL発行」の4ボタン。各ボタンは追客操作を開始する

AIプロファイル表示（NeedsTags）

右側のメインペインのタブの一つとして、AIヒアリング結果から構造化された NeedsTags を表示する。データモデルの needs_tags フィールドに対応し、以下の3カテゴリで色分け表示する:

  • 🔴 dealbreaker（絶対NG条件）: 1階不可、築30年以上不可 など
  • 🟢 must（必須条件）: 駅徒歩10分以内、2LDK以上 など
  • 🟡 nice_to_have（あれば嬉しい条件）: オートロック、宅配ボックス など

NeedsProfile コンポーネントがこの表示を担当する。ヒアリング未完了の顧客には「AIプロファイル未取得」のプレースホルダーを表示する。

タイムライン

右側メインペインの「概要」タブにタイムラインを表示。Timeline + TimelineEvent コンポーネントで、顧客に紐づくすべてのイベントを時系列逆順で描画する:

  • 反響受信、ヒアリングURL送付、ヒアリング完了、フェーズ遷移、通話記録、メール送信、LINE送受信、来店記録、メモ追加 など
  • 各イベントはアイコン＋日時＋概要の1行で表示。クリックで詳細展開

CSS設計

CRM画面のCSS設計は、既存のデザインシステム（frontend/src/design-system/tokens.css）を継承しつつ、CRM固有の要件に対応するCSS変数を追加する方針をとる。

CRM専用CSS変数

:root {
  /* CRM レイアウト */
  --crm-sidebar-width: 220px;
  --crm-content-max-width: 1200px;
  --crm-kanban-column-width: 280px;

  /* CRM カラー（既存トークン継承 + CRM拡張） */
  --crm-phase-inquiry: #e3f2fd;
  --crm-phase-hearing: #fff3e0;
  --crm-phase-visit: #e8f5e9;
  --crm-phase-contracted: #f3e5f5;
  --crm-phase-lost: #fafafa;
}

モバイル vs デスクトップレイアウト方針

既存のヒアリング画面と CRM画面では完全に別レイアウトを採用する:

項目ヒアリング画面CRM画面
最適化対象モバイル（480px）デスクトップ（1200px）
レイアウト既存 Layout.tsxCrmLayout（独自）
サイドバーなし固定220px（モバイルではハンバーガー化）
コンテンツ幅max-width: 480pxmax-width: 1200px
ユースケース顧客がスマホで回答営業担当がPCで操作

CRM画面では Layout.tsx を変更せず、CrmLayout コンポーネントが CrmApp.tsx 内でフルワイドモードのレイアウトを提供する。カンバンビューは横スクロール対応とし、カラム数が多い場合もデスクトップで快適に操作できるようにする。
