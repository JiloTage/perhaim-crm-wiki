Webhook設計詳細

最終更新: 2026-03-18
Per Heim CRM v3.2におけるWebhook受信エンドポイントの詳細設計。不動産ポータルサイト（SUUMO・HOME'S等）からの反響メール転送Webhook、LINE Messaging API Webhook、ヒアリング完了通知、カレンダーイベント通知の4種のWebhookについて、リクエスト/レスポンス仕様・認証・署名検証・メールパーサー・リトライポリシー・セキュリティを定義する。

Webhook設計詳細は、CRM API設計で概要を示したWebhookエンドポイント群の完全な実装仕様書である。Per Heim CRMでは、外部サービスからのイベント通知をWebhookとして受信し、CRMデータへ自動変換する仕組みを採用している。外部連携設計で定義されたLINE Messaging API連携・Gmail反響取込の各フローにおいて、本設計のWebhookがデータ取込の入口として機能する。

すべてのWebhookはセッション認証ではなくWebhook Secret（またはプラットフォーム固有の署名検証）で認証され、冪等性を保証し、デッドレターキューによるエラーハンドリングを備える。本設計はPer Heim CRM v3.2 統合設計書のPhase 4（外部連携）に位置づけられる。

Webhookエンドポイント一覧

Per Heim CRMは以下の4種のWebhookエンドポイントを提供する。すべてのエンドポイントは /v1/webhook/ プレフィックス配下に配置され、CRM APIとは異なる認証方式（Webhook Secret / プラットフォーム署名検証）を採用する。

エンドポイント用途送信元認証方式優先度
POST /v1/webhook/inquiry-email 反響メール転送受信 メール転送サービス / Gmail API Webhook Secret（HMAC-SHA256） Phase 4 MVP
POST /v1/webhook/line-message LINE受信メッセージ通知 LINE Messaging API X-Line-Signature（HMAC-SHA256） Phase 4
POST /v1/webhook/hearing-complete AIヒアリング完了通知 Per Heim AIヒアリングシステム 内部Webhook Secret Phase 3
POST /v1/webhook/calendar-event カレンダーイベント変更通知 Google Calendar API（Pub/Sub） Google Pub/Sub トークン検証 Phase 5

各エンドポイントの共通仕様:

  • Content-Type: application/json
  • レスポンス形式: 成功時は 200 OK、認証失敗時は 401 Unauthorized、バリデーション失敗時は 400 Bad Request
  • タイムアウト: 30秒以内にレスポンスを返却。重い処理は非同期キューに委譲
  • 冪等性: すべてのWebhookは idempotency_key による重複排除を実装

反響メール転送Webhook

反響メール転送Webhookは、不動産ポータルサイト（SUUMO・HOME'S・at home等）からの反響メールを受信し、CRMの反響レコード（crm_inquiries）に自動変換するエンドポイントである。外部連携設計で定義されたGmail反響取込フローの受信口として機能する。

エンドポイント仕様

項目内容
メソッドPOST
パス/v1/webhook/inquiry-email
認証リクエストヘッダ X-Webhook-Signature にHMAC-SHA256署名を付与
Content-Typeapplication/json

リクエスト形式

{
  "raw_email": "From: suumo-noreply@...\nTo: ...\nSubject: ...\n\n本文...",
  "source": "suumo",
  "received_at": "2026-03-18T10:30:00Z",
  "idempotency_key": "msg_abc123def456"
}
  • raw_email（必須・string）: メールの生データ（ヘッダ + 本文）
  • source（必須・string）: メール送信元識別子（suumo, homes, athome, unknown）
  • received_at（任意・ISO 8601）: メール受信日時。省略時はサーバ受信時刻
  • idempotency_key（必須・string）: 冪等性保証用キー。Gmail Message-IDを推奨

レスポンス形式

成功時（200 OK）:

{
  "inquiry_id": "inq_20260318_abc123",
  "status": "new",
  "parsed": true,
  "customer_name": "山田太郎",
  "source": "suumo"
}

パース失敗時（200 OK）:

{
  "inquiry_id": "inq_20260318_def456",
  "status": "manual_review",
  "parsed": false,
  "parse_error": "unknown_format",
  "source": "unknown"
}

認証失敗時（401 Unauthorized）:

{
  "error": {
    "code": "WEBHOOK_AUTH_FAILED",
    "message": "Invalid webhook signature"
  }
}

Webhook Secret認証

リクエストの正当性を検証するため、HMAC-SHA256署名を使用する:

  1. テナント設定画面でWebhook Secretを生成・登録する
  2. 送信側はリクエストボディ全体をWebhook SecretでHMAC-SHA256署名し、X-Webhook-Signature ヘッダに sha256=<hex_digest> 形式で付与する
  3. 受信側は同じSecretでリクエストボディのHMACを計算し、ヘッダの値とタイミングセーフ比較で一致を検証する
# 署名検証の疑似コード
import hmac, hashlib

def verify_webhook_signature(payload: bytes, signature: str, secret: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), payload, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)

メールパーサー仕様

反響メール転送Webhookで受信した raw_email は、ポータルサイト別パーサーによって構造化データに変換される。パーサーは crm/inquiry.py に実装され、正規表現パターンマッチで各フィールドを抽出する。

SUUMO形式

SUUMOからの反響メールは以下の構造を持つ:

抽出項目正規表現パターン必須
顧客名お名前[：:]\s*(.+)Yes
電話番号電話番号[：:]\s*([\d\-]+)No
メールアドレスメール[：:]\s*([\w\.\-]+@[\w\.\-]+)No
問い合わせ物件物件名[：:]\s*(.+)No
メッセージお問い合わせ内容[：:]\s*([\s\S]+?)(?=---|\z)No

HOME'S形式

HOME'Sからの反響メールは表形式のレイアウトを使用する:

抽出項目正規表現パターン必須
顧客名【お客様名】\s*(.+)Yes
電話番号【電話番号】\s*([\d\-]+)No
メールアドレス【メールアドレス】\s*([\w\.\-]+@[\w\.\-]+)No
問い合わせ物件【物件名】\s*(.+)No
メッセージ【問い合わせ内容】\s*([\s\S]+?)(?=【|\z)No

パース処理フロー

  1. source フィールドに基づき対応するパーサーを選択
  2. source が unknown の場合、メール本文の特徴から自動判別を試みる
  3. 各パーサーが正規表現パターンマッチで項目を抽出
  4. 顧客名の抽出に失敗した場合、パース失敗と判定
  5. 電話番号・メールアドレスは正規化処理（ハイフン除去、全角→半角変換）を適用

パース失敗時のフォールバック

パーサーが必須項目の抽出に失敗した場合、以下のmanual_reviewフォールバックが発動する:

  • 反響レコードを status: "manual_review" で作成
  • raw_email をそのまま保存し、担当者が手動で確認・入力できるようにする
  • CRM受信トレイ画面に「要手動確認」バッジを表示
  • 担当者への通知（Slack / メール）を送信
  • パース失敗率が閾値（10%）を超えた場合、パーサー改善アラートを発報

LINE Webhook

LINE WebhookはLINE Messaging APIからの受信メッセージ通知を処理するエンドポイントである。外部連携設計のLINE連携フローに従い、テナントごとに設定されたLINE公式アカウントからのWebhookイベントを受信する。

エンドポイント仕様

項目内容
メソッドPOST
パス/v1/webhook/line-message?tenant_id={tenant_id}
認証X-Line-Signature ヘッダによるHMAC-SHA256署名検証
Content-Typeapplication/json

リクエスト形式

LINE Messaging APIから送信される標準的なWebhookイベント形式:

{
  "destination": "Uxxxxxxxxxxxx",
  "events": [
    {
      "type": "message",
      "message": {
        "type": "text",
        "id": "468XXXXXX",
        "text": "物件について問い合わせたいです"
      },
      "source": {
        "type": "user",
        "userId": "U1234567890abcdef"
      },
      "replyToken": "nHuyWi...",
      "timestamp": 1711111111111
    }
  ]
}

署名検証（X-Line-Signature）

LINE Webhookの認証にはLINE独自の署名検証を使用する:

  1. tenant_id クエリパラメータからテナントを特定
  2. テナント設定からChannel Secretを取得
  3. リクエストボディ全体をChannel SecretでHMAC-SHA256署名し、Base64エンコード
  4. X-Line-Signature ヘッダの値とタイミングセーフ比較で検証
# LINE署名検証の疑似コード
import hmac, hashlib, base64

def verify_line_signature(body: bytes, signature: str, channel_secret: str) -> bool:
    expected = base64.b64encode(
        hmac.new(channel_secret.encode(), body, hashlib.sha256).digest()
    ).decode()
    return hmac.compare_digest(expected, signature)

メッセージ種別ハンドリング

イベントタイプメッセージ種別処理内容
messagetextテキスト本文を crm_communications に記録。line_user_id → customer_id マッピングで顧客特定
messageimage画像URLを保存。Content APIで画像データ取得しCloud Storageに保管
messagestickerスタンプ情報を記録（パッケージID + スタンプID）
messagefileファイルURLを保存。Content APIで取得しCloud Storageに保管
follow-友だち追加イベント。新規 line_user_id を記録、既存顧客とのマッチングを試行
unfollow-ブロックイベント。顧客の line_status を blocked に更新

顧客マッピング

line_user_id から CRM顧客への紐づけフロー:

  1. line_user_id → customer_id マッピングテーブルを検索
  2. マッチあり → 該当顧客のコミュニケーション記録に追加
  3. マッチなし → 未紐づけメッセージとして crm_communications に保存。CRM画面で担当者が手動紐づけ

1つのLINE公式アカウントを複数店舗で共有するケースにも対応するため、マッピングテーブルには tenant_id + line_user_id をキーとして管理する。

リトライ・エラーハンドリング

Webhook処理の信頼性を担保するため、冪等性保証・リトライポリシー・デッドレターキューの3層構造でエラーハンドリングを実装する。

冪等性保証

すべてのWebhookリクエストは冪等性キー（idempotency_key）により重複排除される:

  • 反響メール: Gmail Message-ID をキーとして使用
  • LINE: events[].message.id をキーとして使用
  • ヒアリング完了: session_id をキーとして使用
  • カレンダー: event_id + updated_at をキーとして使用

冪等性キーはFirestoreに24時間保持し、同一キーのリクエストには前回のレスポンスを即時返却する。

リトライポリシー

Webhook送信元からのリトライ、およびCRM側の内部リトライの両方に対応する:

送信元リトライへの対応（受信側として）:

  • 冪等性チェックにより、同一リクエストの再処理を防止
  • 200 OK を速やかに返却し、送信元のリトライを抑制

CRM内部リトライ（Firestoreへの書き込み失敗時など）:

項目設定値
最大リトライ回数3回
初回待機時間1秒
バックオフ方式指数バックオフ + ジッター
最大待機時間30秒
リトライ対象5xx系エラー、タイムアウト、一時的なDB接続エラー
リトライ除外4xx系エラー（バリデーション失敗、認証失敗）
 
# 指数バックオフ計算
import random

def calculate_backoff(attempt: int, base: float = 1.0, max_wait: float = 30.0) -> float:
    wait = min(base * (2 ** attempt), max_wait)
    jitter = random.uniform(0, wait * 0.1)
    return wait + jitter

# attempt 0: 1.0〜1.1秒, attempt 1: 2.0〜2.2秒, attempt 2: 4.0〜4.4秒

デッドレターキュー（DLQ）

最大リトライ回数を超えて処理に失敗したメッセージはデッドレターキューに投入される:

  • 保存先: Firestoreの crm_webhook_dlq コレクション
  • 保持期間: 30日間
  • 保存データ: リクエストボディ全体、エラー情報、リトライ履歴、受信日時
  • 管理画面: 管理者がDLQメッセージを確認し、手動で再処理 or 破棄を選択

セキュリティ

Webhookエンドポイントは外部からアクセス可能なため、多層防御のセキュリティ設計を採用する。

Webhook Secret検証

  • 各テナント・各Webhookタイプごとに固有のWebhook Secretを発行
  • SecretはSecret Manager（Google Cloud Secret Manager）で管理
  • 定期的なSecretローテーション（90日ごと推奨）をサポート
  • ローテーション中は新旧2つのSecretを同時に受け付ける移行期間を設ける

HMAC署名

すべてのWebhookはHMAC-SHA256署名で正当性を検証する:

  • 反響メール: X-Webhook-Signature: sha256=<hex_digest>
  • LINE: X-Line-Signature: <base64_digest>（LINE仕様に準拠）
  • ヒアリング完了: X-Webhook-Signature: sha256=<hex_digest>
  • カレンダー: Google Pub/Subのトークンベース検証

署名検証は必ずタイミングセーフ比較（hmac.compare_digest）を使用し、タイミング攻撃を防止する。

IPホワイトリスト

Webhookエンドポイントへのアクセスを送信元IPアドレスで制限する:

Webhookホワイトリスト対象管理方式
反響メールメール転送サービスのIPレンジCloud Armor ルール
LINELINE Platform IPレンジ（公式ドキュメント参照）Cloud Armor ルール
ヒアリング完了内部ネットワーク / VPCVPCファイアウォール
カレンダーGoogle Pub/Sub IPレンジCloud Armor ルール

レート制限

不正アクセスやDoS攻撃を防止するため、Webhookエンドポイントにレート制限を適用する:

エンドポイントレート制限バースト許容
/v1/webhook/inquiry-email100 req/min per tenant200 req/min
/v1/webhook/line-message500 req/min per tenant1000 req/min
/v1/webhook/hearing-complete50 req/min per tenant100 req/min
/v1/webhook/calendar-event100 req/min per tenant200 req/min

レート制限超過時は 429 Too Many Requests を返却し、Retry-After ヘッダで再試行可能時刻を通知する。

その他のセキュリティ対策

  • TLS 1.2以上を必須とし、平文通信は一切許可しない
  • リクエストボディサイズの上限制限（10MB）を設定
  • タイムスタンプ検証: リクエストのタイムスタンプが5分以上古い場合は拒否（リプレイ攻撃防止）
  • すべてのWebhookアクセスをCloud Loggingに記録し、監査証跡を維持
  • raw_email に含まれるHTMLタグはサニタイズしてからDBに保存
