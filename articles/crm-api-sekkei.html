CRM API設計

最終更新: 2026-03-18
Per Heim CRM v3.2の全APIエンドポイント設計。既存の/v1/tenant/*と共存するCRM専用プレフィックス配下に、顧客管理・タスク管理・追客・反響管理・分析・カンバンの各APIを定義し、RESTful設計原則に基づくリソース指向インターフェースを提供する。

CRM API設計は、Per Heim CRM v3.2 統合設計書で定義されたハイブリッド拡張アーキテクチャの中核を成すバックエンドインターフェース仕様である。既存のAIヒアリングAPI（/v1/tenant/*）を一切変更せず、新規のCRM専用プレフィックス（/v1/tenant/crm/*）配下に全エンドポイントを配置する。

本設計では、CRMデータモデル設計で定義されたFirestoreコレクションに対するCRUD操作に加え、フェーズ遷移・顧客マージ・追客記録・反響取込Webhook・分析集計・カンバン操作など、CRM業務に特化したドメインアクションを提供する。すべての変更系操作はタイムラインイベントを副作用として生成し、顧客活動の完全な追跡を可能にする。

APIプレフィックスとバージョニング

Per Heimの既存APIは /v1/tenant/* プレフィックス配下で招待管理・セッション管理・アーティファクト取得を提供している。CRM APIはこの既存構造と完全に共存する形で、専用のサブプレフィックスを追加する。

カテゴリプレフィックス対象状態
既存/v1/tenant/*招待管理、AIヒアリング変更なし
CRM新規/v1/tenant/crm/*顧客・タスク・追客・分析新規追加
Webhook/v1/webhook/*反響メール受信新規追加

バージョニング方針:

  • URLパスベース: /v1/ をプレフィックスとし、メジャーバージョンのみをURLに含める
  • 後方互換性: 同一バージョン内ではフィールド追加のみ許容し、削除・型変更は次バージョンで実施
  • 認証方式: 既存のセッションCookie認証を共有。Webhookのみ別途Webhook Secretによる認証
  • テナント分離: すべてのリクエストはセッションから tenant_id を解決し、データアクセスをテナント単位で制限

顧客管理API

顧客管理APIはCRMデータモデルの customers コレクションに対するCRUD操作と、ドメイン固有のアクション（フェーズ遷移、担当者変更、ヒアリングURL発行、マージ、重複検索）を提供する。

CRUD操作

メソッドエンドポイント説明
GET/v1/tenant/crm/customers顧客一覧取得（ページネーション・フィルタ対応）
GET/v1/tenant/crm/customers/{customer_id}顧客詳細（タイムライン・タスク・コミュニケーション含む）
POST/v1/tenant/crm/customers顧客新規作成
PATCH/v1/tenant/crm/customers/{customer_id}顧客情報更新
DELETE/v1/tenant/crm/customers/{customer_id}顧客削除（論理削除）

顧客一覧のクエリパラメータ:

  • phase: フェーズでフィルタ（new, responding, hearing, visit, negotiation, contracted, lost）
  • search: 名前・電話番号・メールアドレスでの検索
  • assigned_to: 担当者IDでフィルタ
  • store_id: 店舗IDでフィルタ
  • sort: ソート指定（updated_at, created_at, name）
  • page, limit: ページネーション（デフォルト: page=1, limit=50）

顧客詳細レスポンスは、顧客基本情報に加えてタイムライン・タスク・コミュニケーション履歴・AIヒアリング成果物を一括で返却する。これにより、顧客カード画面を1リクエストで描画可能にする。

ドメインアクション

メソッドエンドポイント説明副作用
POST /v1/tenant/crm/customers/{id}/phase フェーズ遷移 タイムラインイベント + 自動タスク生成
POST /v1/tenant/crm/customers/{id}/assign 担当者変更 タイムラインイベント
POST /v1/tenant/crm/customers/{id}/hearing-invite AIヒアリングURL発行 既存lead作成 + customer.lead_ids追加
POST /v1/tenant/crm/customers/{id}/merge 顧客マージ タイムライン統合 + タスク移行
GET /v1/tenant/crm/customers/{id}/duplicates 重複候補検索 なし（読み取り専用）

フェーズ遷移はCRM業務の中核操作であり、遷移時に以下の処理が自動実行される:

  1. 顧客の phase フィールドを更新
  2. タイムラインイベント（phase_changed）を記録
  3. 遷移先フェーズに応じた自動タスクを生成（例: hearing → 「ヒアリング結果確認」タスク）

重複候補検索は、電話番号・メールアドレスの正規化後の検索用ハッシュ（*_search_hash）を使い、平文一致検索は行わない。これはCRMデータモデル設計のPII暗号化方針に準拠する。

ヒアリングURL発行は既存の invite_sales のlead作成APIをラッパーとして呼び出し、CRM側の customer.lead_ids に追加する。これにより既存AIヒアリング機能を一切変更せずにCRMから利用可能にする。

タスク管理API

タスク管理APIは、営業担当者の日常業務を管理するためのエンドポイント群である。タスクは必ず顧客に紐づき、フェーズ遷移時の自動生成タスクと営業担当者が手動作成するタスクの両方を統一的に管理する。

メソッドエンドポイント説明
GET/v1/tenant/crm/tasksタスク一覧（フィルタ: status, assigned_to, due_date, customer_id）
POST/v1/tenant/crm/tasksタスク作成
PATCH/v1/tenant/crm/tasks/{task_id}タスク更新（ステータス・タイトル・期限・優先度）
POST/v1/tenant/crm/tasks/{task_id}/completeタスク完了（副作用: タイムラインイベント）

タスク作成時のリクエストボディ:

  • customer_id（必須）: 紐づけ先の顧客ID
  • title（必須）: タスクタイトル
  • task_type: タスク種別（call, email, visit, follow_up, other）
  • priority: 優先度（high, medium, low）
  • due_date: 期限日
  • description: 詳細説明

追客API

追客APIは、顧客とのコミュニケーション記録を管理するエンドポイント群である。通話記録・メール送信・LINE送信の3チャネルに対応し、すべての操作はタイムラインイベントとして記録される。

メソッドエンドポイント説明
GET/v1/tenant/crm/customers/{id}/communicationsコミュニケーション記録一覧
POST/v1/tenant/crm/customers/{id}/call-log通話記録追加
POST/v1/tenant/crm/customers/{id}/emailメール送信
POST/v1/tenant/crm/customers/{id}/lineLINE送信

通話記録

通話記録は営業担当者が手動で登録する。リクエストボディには以下を含む:

  • direction: 発信（outbound）/ 着信（inbound）
  • call_result: 通話結果（connected, no_answer, busy, voicemail）
  • call_memo: 通話メモ（任意）
  • call_duration_seconds: 通話時間（秒、任意）

メール送信

メール送信APIはテンプレート対応で、template_id を指定すると事前定義されたテンプレートに顧客情報を差し込んで送信する。テンプレート未指定時は subject と body をそのまま使用する。

LINE送信

LINE送信は手動送信のみ対応（Phase 4の設計範囲）。顧客の line_id が登録されている場合にのみ利用可能。自動メッセージ送信は将来拡張として検討する。

反響管理API

反響管理APIは、不動産ポータルサイト（SUUMO、HOME'S等）からの反響メールを受信・パースし、CRMの顧客データに変換するためのエンドポイント群である。

メソッドエンドポイント説明
GET/v1/tenant/crm/inquiries反響一覧（受信トレイ、フィルタ: status, source）
POST/v1/tenant/crm/inquiries/{id}/link反響→顧客紐づけ（customer_id省略時は新規顧客自動作成）
POST/v1/tenant/crm/inquiries/{id}/ignore反響無視

メール転送Webhook

反響メールの取込はWebhook経由で行う。不動産ポータルからの反響メールをメール転送サービス経由で受信し、自動的にパースして反響レコードを生成する。

項目内容
エンドポイントPOST /v1/webhook/inquiry-email
認証Webhook Secret（セッション認証ではない）
リクエストボディ{ raw_email: string, source: string }
レスポンス{ inquiry_id: string }

Webhookの詳細な設計（リトライポリシー、署名検証、メールパーサー仕様など）はWebhook設計詳細を参照。

分析API

分析APIは、CRMに蓄積されたデータから営業KPI・ファネル分析・リードタイム分析・月次レポートを提供する読み取り専用のエンドポイント群である。

メソッドエンドポイント説明
GET/v1/tenant/crm/analytics/dashboardダッシュボードKPI
GET/v1/tenant/crm/analytics/funnelファネルデータ
GET/v1/tenant/crm/analytics/lead-timeリードタイム分析
GET/v1/tenant/crm/analytics/monthly-report月次レポート

ダッシュボードKPI

期間指定（7d / 30d / 90d）で以下のKPIを返却する:

  • total_customers: 総顧客数
  • new_inquiries: 新規反響数
  • hearing_completion_rate: ヒアリング完了率
  • visit_rate: 来店率
  • contract_rate: 成約率
  • avg_lead_time_days: フェーズ別平均リードタイム

ファネルデータ

compare_perheim=true パラメータにより、全体ファネルとPer Heim経由ファネルのデュアルファネル比較が可能。Per Heimの効果測定に活用する。

リードタイム分析

フェーズ間の平均遷移日数を返却する。各フェーズ遷移ペア（from_phase → to_phase）ごとの avg_days を配列で提供し、ボトルネックとなっているフェーズの特定に活用する。

月次レポート

指定月（YYYYMM形式）の包括的なレポートデータを返却する。ダッシュボードKPIの月次集計に加え、担当者別・店舗別の実績データを含む。

カンバンAPI

カンバンAPIは、フロントエンドのカンバンボードUIに最適化されたエンドポイントを提供する。フェーズ別に顧客をグループ化し、ドラッグ&ドロップによるカード移動（= フェーズ遷移）をサポートする。

メソッドエンドポイント説明
GET/v1/tenant/crm/kanbanカンバンデータ取得（フェーズ別顧客グループ）
POST/v1/tenant/crm/kanban/moveカード移動（= フェーズ遷移）

カンバンデータ取得は、各フェーズをカラムとして顧客サマリーを返却する:

  • store_id: 店舗でフィルタ（任意）
  • assigned_to: 担当者でフィルタ（任意）

レスポンスは columns 配列で、各カラムに phase（フェーズ名）、customers（顧客サマリー配列）、count（件数）を含む。

カード移動（POST /v1/tenant/crm/kanban/move）は内部的に顧客管理APIのフェーズ遷移（POST /v1/tenant/crm/customers/{id}/phase）と同等の処理を実行する。カンバンUI専用のエンドポイントとして、customer_id と to_phase を直接受け取る簡易インターフェースを提供する。

リクエスト処理フロー

すべてのCRM APIリクエストは以下の共通フローで処理される:

sequenceDiagram
    participant Client as フロントエンド
    participant Auth as 認証ミドルウェア
    participant Router as CRM Router
    participant Service as CRM Service
    participant FS as Firestore
    participant TL as タイムライン

    Client->>Auth: リクエスト（セッションCookie）
    Auth->>Auth: tenant_id / employee_id 解決
    Auth->>Router: 認証済みリクエスト
    Router->>Service: ビジネスロジック実行

    alt 変更系操作
        Service->>FS: データ更新
        Service->>TL: タイムラインイベント生成
        TL->>FS: イベント永続化
    else 読み取り系操作
        Service->>FS: データ取得
    end

    Service->>Router: レスポンス
    Router->>Client: JSON レスポンス

認証ミドルウェアは既存のセッションCookie認証基盤を共有し、CRM APIでは追加で3ロール権限マトリクス（管理者 / マネージャー / 担当者）によるアクセス制御を実施する。

セキュリティと共通仕様

PII取り扱い

顧客の電話番号・メールアドレスはAPIでは平文で受け渡しするが、永続化時に以下の処理が行われる:

  • 暗号化列への保存（AES-256-GCM）
  • 検索用ハッシュ（phone_search_hash, email_search_hash）の生成
  • 重複検知は正規化後のハッシュを使用し、平文一致検索は行わない

エラーレスポンス形式

すべてのAPIは統一されたエラーレスポンス形式を返却する:

{
  "error": {
    "code": "CUSTOMER_NOT_FOUND",
    "message": "指定された顧客が見つかりません",
    "details": {}
  }
}

ページネーション

一覧系APIは共通のページネーションパラメータ（page, limit）を受け付け、レスポンスには total（総件数）と page（現在ページ）を含む。
