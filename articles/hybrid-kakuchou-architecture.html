ハイブリッド拡張アーキテクチャ

最終更新: 2026-03-18
Per Heim CRM v3.2 統合におけるハイブリッド拡張アーキテクチャの全体設計。既存AIヒアリング機能を一切変更せず、CRM機能を独立モジュールとして追加し、Firestoreを共有データ基盤とする方針を定義する。

モジュール分離方針

ハイブリッド拡張アーキテクチャの核心は、既存コードへの変更をゼロに保ちながら、CRM機能を完全に独立したモジュールとして追加する点にある。バックエンドは src/server/crm/、フロントエンドは frontend/src/crm/ をルートディレクトリとし、既存の invite_sales/ やページコンポーネントとは物理的に分離する。

フロントエンド構造

フロントエンドも同様に、既存の pages/（SalesConsole, CompanyConsole）は一切変更せず、frontend/src/crm/ 配下に独立したCRMアプリケーションを構築する:

  • crm/CrmApp.tsx — CRMルートコンポーネント（サイドバー + ルーティング）
  • crm/pages/ — Dashboard, CustomerKanban, CustomerList, CustomerDetail, TaskList, Analytics, InquiryInbox, Settings
  • crm/components/ — KanbanBoard, Timeline, CustomerCard, PhaseTag, CommunicationPanel, FunnelChart 等
  • crm/hooks/ — useCrmApi, useCustomers, useTasks, useAnalytics
  • crm/types.ts — CRM型定義

ルーティングは App.tsx に /crm パスを追加し、RequireTenantAuth ガードの内側に配置する。これにより認証済みテナントのみがCRM画面にアクセスできる。

既存コードとの接続ポイント

CRMモジュールが既存の invite_sales と接続する方法は、読み取り専用参照、イベントフック、認証共有の3つに限定される。この制約により、既存機能の安定性を保証する。

読み取り専用参照

CRMの repository.py から既存の get_invite_sales_repo() を参照し、以下のデータを読み取り専用で取得する:

既存データCRM側の用途アクセス方法
leads顧客レコードへの紐づけlead_id → customer.lead_ids
artifactsAIプロファイル表示hearing_profile への転記
sessionsヒアリング状態の照会hearing_status の確認
tenant_accountsテナント情報の共有既存認証基盤経由
store_registry店舗情報の共有既存認証基盤経由
tenant_employees従業員・担当者情報既存認証基盤経由 + role拡張

イベントフック

既存の complete_session() 内部にイベントフックを追加し、AIヒアリング完了時にCRM側の on_hearing_complete() を呼び出す。これにより以下の自動処理が実行される:

  • 顧客フェーズの自動遷移（hearing_active → hearing_done）
  • 担当者への完了通知
  • AIプロファイルの hearing_profile への転記
  • タイムラインへのイベント記録

認証共有

認証は既存の tenant_session_cookie を共有するが、CRM APIの操作者解決は X-Tenant-Employee-Id ヘッダーを必須とする。Phase 0 で resolve_tenant_actor() 公開ヘルパーを追加し、既存の _require_tenant_session() と _require_operator_employee() をラップしてCRM側から再利用する。詳細は認証・権限詳細を参照。

認証・権限設計

CRM統合にあたり、既存の2階層認証（company | store）を拡張し、3ロールの業務権限レイヤーを追加する。ログイン境界と業務権限を分離することで、既存認証を壊さずにきめ細やかなアクセス制御を実現する。

CrmActorの解決フロー

CRM APIはCookie単独では操作者を確定しない。tenant_session_cookie（ログイン境界: company/store）と X-Tenant-Employee-Id（操作者識別）を組み合わせて CrmActor オブジェクトを構築する。

重要な設計判断: Phase 0 で invite_sales 側に公開ヘルパー（resolve_tenant_actor()）を追加し、privateメソッドの直接呼び出しを避ける。assigned_only スコープの判定は employee_id ベースで行い、Cookie単独では許可しない。

外部連携アーキテクチャ

CRM v3.2 は4つの外部サービスと連携する。すべての連携は src/server/integrations/ モジュールに集約され、CRMビジネスロジックとは疎結合に保たれる。詳細は外部連携詳細を参照。

LINE Messaging API 連携

テナント設定画面でLINE Channel Access Token / Channel Secretを登録し、Webhook URL（/v1/webhook/line-message?tenant_id=xxx）で受信メッセージを取得する。受信メッセージは crm_communications と crm_timeline に記録され、CRM画面からの返信はLINE Reply/Push APIで送信する。

複数店舗で1つのLINE公式アカウントを共有するケースに対応するため、line_user_id → customer_id のマッピングテーブルで管理する。

Gmail 反響取込

SUUMO/HOME'S/at home からの反響メールを専用Gmailアドレスに転送し、Gmail API経由で取得・パースする。パース対象は顧客名、電話番号、メール、問い合わせ物件、メッセージで、パース失敗時は status: "manual_review" で担当者に確認を促す。

MVPではGmail API Polling（1分間隔）方式を採用し、Cloud Schedulerで定期実行する。リアルタイム性が必要な場合はPub/Sub連携に移行可能。

Google Calendar 連携

CRM画面での来店予約登録をGoogle Calendar APIで同期する。カレンダー変更のWebhook通知により、来店予約 → 来店済み（visit_reserved → visited）のフェーズ自動遷移を実現する。

CSVインポート

フロントエンドでのファイル選択 → プレビュー → カラムマッピング → ドライラン → 実行の5ステップUIを提供する。バックエンドはパース → バリデーション → 重複検知 → Bulk Insertの処理パイプラインを担う。

Firestore設計方針

既存システムは invite_sales_state/default という単一ドキュメントにすべてのデータを格納する方式だが、CRMのデータ量増加に耐えられない。CRM機能ではFirestoreネイティブコレクション（個別コレクション方式）に移行する。

コレクション構成

コレクションパス用途備考
invite_sales_state/default既存データ（変更しない）レガシー単一ドキュメント
crm_customers/{customer_id}顧客マスタ新規・1顧客1ドキュメント
crm_tasks/{task_id}タスク新規・期限管理対応
crm_timeline/{event_id}タイムラインイベント新規・全イベント時系列
crm_communications/{comm_id}コミュニケーション記録新規・LINE/メール/電話
crm_inquiries/{inquiry_id}反響レコード新規・ポータル連携
crm_analytics/{tenant_id}/monthly/{yyyymm}月次集計データ新規・サブコレクション
crm_settings/{tenant_id}テナントCRM設定新規・LINE/Gmail認証情報等

インデックス設計

CRMの主要クエリパターンに対応するため、以下の複合インデックスを設計する:

  • crm_customers: (tenant_id, phase, updated_at DESC) — カンバンビューのフェーズ別表示
  • crm_customers: (tenant_id, assigned_employee_id, phase) — 担当者別顧客フィルタ
  • crm_customers: (tenant_id, store_id, phase) — 店舗別顧客一覧
  • crm_tasks: (tenant_id, assigned_employee_id, due_date ASC) — 本日のタスクリスト
  • crm_timeline: (customer_id, created_at DESC) — 顧客タイムライン表示
  • crm_communications: (customer_id, created_at DESC) — コミュニケーション履歴
  • crm_inquiries: (tenant_id, status, created_at DESC) — 反響受信トレイ

前提条件

  • CRM機能を有効化する環境では INVITE_SALES_STORAGE_BACKEND=firestore を必須化する
  • local/dev環境はFirestore Emulatorを使用。JSON backendは既存 invite_sales のみを対象とするレガシー運用とし、CRMの正式サポート対象外
  • invite_sales_state/default と CRMコレクション群が同一Firestore上で一貫して参照可能
