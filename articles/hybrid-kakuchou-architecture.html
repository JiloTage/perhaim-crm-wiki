ハイブリッド拡張アーキテクチャ

最終更新: 2026-03-18
Per Heim CRM v3.2 統合におけるハイブリッド拡張アーキテクチャの全体設計。既存AIヒアリング機能を一切変更せず、CRM機能を独立モジュールとして追加し、Firestoreを共有データ基盤とする方針を定義する。

モジュール分離方針

ハイブリッド拡張アーキテクチャの核心は、既存コードへの変更をゼロに保ちながら、CRM機能を完全に独立したモジュールとして追加する点にある。バックエンドは src/server/crm/、フロントエンドは frontend/src/crm/ をルートディレクトリとし、既存の invite_sales/ やページコンポーネントとは物理的に分離する。

フロントエンド構造

フロントエンドも同様に、既存の pages/（SalesConsole, CompanyConsole）は一切変更せず、frontend/src/crm/ 配下に独立したCRMアプリケーションを構築する:

  • crm/CrmApp.tsx — CRMルートコンポーネント（サイドバー + ルーティング）
  • crm/pages/ — Dashboard, CustomerKanban, CustomerList, CustomerDetail, TaskList, Analytics, InquiryInbox, Settings
  • crm/components/ — KanbanBoard, Timeline, CustomerCard, PhaseTag, CommunicationPanel, FunnelChart 等
  • crm/hooks/ — useCrmApi, useCustomers, useTasks, useAnalytics
  • crm/types.ts — CRM型定義

ルーティングは App.tsx に /crm パスを追加し、RequireTenantAuth ガードの内側に配置する。これにより認証済みテナントのみがCRM画面にアクセスできる。

既存コードとの接続ポイント

CRMモジュールが既存の invite_sales と接続する方法は、読み取り専用参照、イベントフック、認証共有の3つに限定される。この制約により、既存機能の安定性を保証する。

読み取り専用参照（公開APIパターン）

レビュー指摘 C1 対応: 従来の「読み取り専用参照」は具体策が不足しており、threading lock競合・不整合データ読み取り・1MBドキュメント全取得のリスクがあった。以下の公開APIパターンで解決する。

CRMバックエンドから既存 invite_sales のデータにアクセスする際は、invite_sales/public_api.py の公開メソッド経由のみとする。_db への直接参照は禁止し、lock内でのdeepcopyにより安全なスナップショットを返す。

公開APIの設計

# invite_sales/public_api.py — Phase 0 で実装
"""invite_sales の公開インターフェース。
CRM バックエンドはこのモジュール経由でのみ invite_sales データにアクセスする。
"""
import copy
from typing import Optional

class InviteSalesPublicAPI:
    def __init__(self, repo):
        self._repo = repo  # InviteSalesRepository

    def get_lead_snapshot(self, tenant_id: str, lead_id: str) -> Optional[dict]:
        """lock 内で deepcopy した lead のスナップショットを返す。
        CRM 側で安全に読み取り可能（threading lock 競合なし）。
        """
        with self._repo._lock:
            lead = self._repo._db.get("leads", {}).get(lead_id)
            return copy.deepcopy(lead) if lead else None

    def get_artifacts_snapshot(self, tenant_id: str, lead_id: str) -> Optional[dict]:
        """AI ヒアリング成果物（hearing_profile, needs_tags）のスナップショット。"""
        with self._repo._lock:
            artifacts = self._repo._db.get("artifacts", {}).get(lead_id)
            return copy.deepcopy(artifacts) if artifacts else None

    def get_session_status(self, tenant_id: str, lead_id: str) -> Optional[str]:
        """ヒアリングセッションの現在状態を返す。"""
        with self._repo._lock:
            sessions = self._repo._db.get("sessions", {})
            for session in sessions.values():
                if session.get("lead_id") == lead_id:
                    return session.get("status")
            return None

イベントフック（Cloud Tasks非同期方式）

レビュー指摘 C2 対応: 従来設計の同期的 on_hearing_complete() 呼び出しは、lock内CRM書き込みでdeadlockリスクがあり、失敗時リトライも未定義だった。Cloud Tasks による非同期フック方式に変更する。

既存の complete_session() のlock外側で、Cloud Tasks に非同期タスクをenqueueする。CRM側は内部エンドポイントでイベントを受信し、独立したトランザクションで処理する。

  • AIプロファイルの hearing_profile への転記
  • タイムラインへのイベント記録

認証共有（セッション埋め込み方式）

レビュー指摘 C3 対応: 従来の X-Tenant-Employee-Id ヘッダーは平文文字列であり、同一テナント内の別従業員IDを自由にセットできるため、なりすましリスクがあった。セッション埋め込み方式に変更する。

認証は既存の tenant_session_cookie を共有する。X-Tenant-Employee-Id ヘッダーは廃止し、ログイン時に employee_id + employee_role をセッショントークンに含める方式に変更する。サーバー側はセッションから employee_id を取得し、ヘッダー偽装の余地を排除する。

CRM では「1ログイン = 1従業員」モデルを採用する（CRM v3.2 の F8-4 per-employee 認証に合致）。Phase 0 で resolve_tenant_actor() を公開ヘルパーとして実装し、既存の _require_tenant_session() と _require_operator_employee() をラップしてCRM側から再利用する。詳細は認証・権限詳細を参照。

認証・権限設計

CRM統合にあたり、既存の2階層認証（company | store）を拡張し、3ロールの業務権限レイヤーを追加する。ログイン境界と業務権限を分離することで、既存認証を壊さずにきめ細やかなアクセス制御を実現する。

CrmActorの解決フロー

CRM APIは tenant_session_cookie に埋め込まれた employee_id と employee_role を使って CrmActor オブジェクトを構築する。ヘッダーによる操作者指定は廃止され、セッショントークンが唯一の操作者識別手段となる。

重要な設計判断:

  • Phase 0 で invite_sales 側に公開ヘルパー（resolve_tenant_actor()）を追加し、privateメソッドの直接呼び出しを避ける
  • X-Tenant-Employee-Id ヘッダーは完全に廃止。セッショントークンから employee_id を取得することで、なりすましリスクを排除する
  • assigned_only スコープの判定は employee_id ベースで行い、Cookie単独では許可しない
  • ログイン時にセッションへ employee_id + employee_role を含める実装を Phase 0 の resolve_tenant_actor() に統合する

外部連携アーキテクチャ

CRM v3.2 は4つの外部サービスと連携する。すべての連携は src/server/integrations/ モジュールに集約され、CRMビジネスロジックとは疎結合に保たれる。詳細は外部連携詳細を参照。

LINE Messaging API 連携

テナント設定画面でLINE Channel Access Token / Channel Secretを登録し、Webhook URL（/v1/webhook/line-message?tenant_id=xxx）で受信メッセージを取得する。受信メッセージは crm_communications と crm_timeline に記録され、CRM画面からの返信はLINE Reply/Push APIで送信する。

複数店舗で1つのLINE公式アカウントを共有するケースに対応するため、line_user_id → customer_id のマッピングテーブルで管理する。

Gmail 反響取込

SUUMO/HOME'S/at home からの反響メールを専用Gmailアドレスに転送し、Gmail API経由で取得・パースする。パース対象は顧客名、電話番号、メール、問い合わせ物件、メッセージで、パース失敗時は status: "manual_review" で担当者に確認を促す。

MVPではGmail API Polling（1分間隔）方式を採用し、Cloud Schedulerで定期実行する。リアルタイム性が必要な場合はPub/Sub連携に移行可能。

Google Calendar 連携

CRM画面での来店予約登録をGoogle Calendar APIで同期する。カレンダー変更のWebhook通知により、来店予約 → 来店済み（visit_reserved → visited）のフェーズ自動遷移を実現する。

CSVインポート

フロントエンドでのファイル選択 → プレビュー → カラムマッピング → ドライラン → 実行の5ステップUIを提供する。バックエンドはパース → バリデーション → 重複検知 → Bulk Insertの処理パイプラインを担う。

Firestore設計方針

既存システムは invite_sales_state/default という単一ドキュメントにすべてのデータを格納する方式だが、CRMのデータ量増加に耐えられない。CRM機能ではFirestoreネイティブコレクション（個別コレクション方式）に移行する。
