CRM実装フェーズ計画

最終更新: 2026-03-18
Per Heim CRM v3.2の段階的実装計画を定義する。Phase 0（基盤準備）からPhase 7（連携・通知・権限）までの全8段階の開発ロードマップに加え、通知設計、分析・レポート設計、非機能要件、リスクと対策、および判断が必要な項目を包括的にカバーする。

通知設計

CRM運用において、適切なタイミングでの通知は営業の反応速度と顧客満足度に直結する。本設計では3段階の通知チャネルとNotificationServiceを中核としたアーキテクチャを定義する。

通知チャネル

レベルチャネル実装方式
即時 ブラウザ通知（Web Push）+ CRM画面内バッジ Service Worker + Firestore onSnapshot
日次 メール Cloud Scheduler + SendGrid / Gmail API
エスカレーション メール + CRM画面 Cloud Scheduler + 条件判定ロジック

通知イベント一覧

通知イベントは大きく即時通知、日次バッチ、エスカレーションの3種に分類される。

即時通知

  • inquiry_received — 新規反響受信時、担当者またはラウンドロビンで割当先に「新規反響: {customer_name} ({source})」を push + badge で通知
  • hearing_completed — AIヒアリング完了時、担当者に「{customer_name}のAIヒアリングが完了しました」を push + badge で通知
  • line_received — LINEメッセージ受信時、担当者に「{customer_name}からLINEメッセージ」を push + badge で通知

日次バッチ

  • daily_task_summary — 毎日09:00 JST に全従業員へ「本日のタスク: {pending_count}件」をメール送信
  • overdue_reminder — 毎日17:00 JST に担当者へ「未完了タスク: {overdue_count}件」をメール + badge で通知

エスカレーション

  • inquiry_unresponded_30min — 反響未対応30分超過でマネージャーにメール + push で通知
  • task_overdue_1day — タスク期限1日超過でマネージャーにメールで通知

通知実装アーキテクチャ

通知の送信はNotificationServiceが一元管理する。即時通知はWeb Push APIとFirestore書き込みで処理し、非同期メール送信はCloud Tasksにエンキューする。定期バッチはCloud Schedulerがトリガーする。

分析・レポート設計

CRM v3.2の分析機能は、デュアルファネルによる全体/Per Heim経由の比較と、ROI訴求指標による効果測定を柱とする。経営者への訴求力を高めるため、「Per Heimを使うと来店率がX%上がる」ことをデータで証明できる設計とする。

デュアルファネル

全体ファネルとPer Heim経由ファネルの2系統を並列表示し、Per Heimの効果を可視化する。

ステージ全体ファネルPer Heim経由ファネル
反響 / ヒアリング依頼45件20件
ヒアリング中 / 完了20件15件
来店予約8件（来店率 17.8%）7件（来店率 35.0%）
成約3件3件

Per Heim経由の来店率は全体の約2倍となり、AIヒアリングの効果を定量的に示すことができる。

ROI訴求指標（RoiMetrics）

RoiMetricsモデルで以下の比較指標を管理する:

  • Per Heim経由: 反響数、来店率、成約率、平均リードタイム
  • 非Per Heim: 同上の指標
  • 差分: 来店率改善（visit_rate_improvement）、成約率改善（contract_rate_improvement）、リードタイム短縮日数（lead_time_reduction_days）

集計方式

  • MVP段階: Firestoreクエリでリアルタイム集計（データ量が少ないため許容）
  • 将来拡張: crm_analytics/{tenant_id}/monthly/{yyyymm} に日次バッチで集計済みデータを保存

分析機能の画面設計とグラフ仕様の詳細は 分析レポート詳細 を参照。

非機能要件

CRM統合にあたり、パフォーマンス、セキュリティ、データ保持の3領域で明確な基準を定める。既存のハイブリッド拡張アーキテクチャの制約を踏まえた設計となっている。

パフォーマンス基準

指標基準値実現手段
画面表示2秒以内Firestoreインデックス + フロントキャッシュ
API応答200ms以内Firestoreクエリ最適化
同時ユーザー100ユーザーCloud Runオートスケール

Firestoreインデックス設計

CRM機能で必要な複合インデックスは firestore.indexes.json に以下を追加する:

  • crm_customers: tenant_id + phase + updated_at (DESC) — カンバン表示用
  • crm_customers: tenant_id + store_id + phase — 店舗別フィルタ用
  • crm_tasks: tenant_id + assigned_employee_id + due_date + status — タスクリスト用
  • crm_timeline: tenant_id + customer_id + created_at (DESC) — タイムライン表示用

インデックス設計の詳細は CRMデータモデル設計 を参照。

項目対応

Phase 0: 基盤準備（3〜5日）

目的: CRMモジュールの骨格を作成し、既存コードに影響なく並行開発できる状態にする。

  • src/server/crm/ ディレクトリ構造作成
  • INVITE_SALES_STORAGE_BACKEND=firestore をCRM前提として有効化
  • local/dev向け Firestore Emulator 起動手順追加
  • CRM用Firestoreリポジトリ基盤（CRUD抽象）
  • crm_api.py ルーター登録（main.py に追加）
  • actor解決ヘルパー追加（tenant_session + X-Tenant-Employee-Id）
  • frontend/src/crm/ ディレクトリ構造作成
  • CrmLayout コンポーネント（サイドバー + コンテンツ）
  • App.tsx に nested route として /crm を追加
  • テスト基盤

成果物: /crm にアクセスすると空のダッシュボードが表示される。

Phase 1: 顧客管理コア（2〜3週間）

目的: 既存leadを「顧客」として管理できるようにする。依存: Phase 0

  • Customer データモデル + Firestore CRUD
  • 既存lead → customer 自動変換ロジック（初回は 1 lead = 1 customer）
  • 顧客一覧API + 顧客一覧画面
  • 顧客詳細API + 顧客カード画面
  • フェーズ遷移API + カンバンビュー
  • タイムラインAPI + タイムライン表示
  • merge UI と暫定重複候補表示
  • 顧客検索・フィルタ
  • ユニットテスト

成果物: カンバンで顧客をフェーズ管理でき、タイムラインが見える。データモデルの詳細は CRMデータモデル設計 を参照。

Phase 2: タスク管理（1〜2週間）

目的: 「本日のタスク」で営業の日次アクションを管理。依存: Phase 1

Phase 3: AIヒアリング統合強化（1〜2週間）

  • 顧客カードへのAIプロファイル統合表示
  • CRM画面からのヒアリングURL発行
  • テスト

成果物: ヒアリング完了が即座にCRMに反映される。

Phase 4: 追客・コミュニケーション（3〜4週間）

目的: CRMから顧客に連絡できるようにする。依存: Phase 1

Phase 5: 反響管理（3〜4週間）

  • 反響受信トレイUI
  • 正規化 + search hash による重複顧客検知

  • サンクスメール自動送信
  • テスト

  • Per Heim経由 vs 非経由 比較

リスクと対策

CRM統合に伴う主要リスクを技術リスクと運用リスクに分類し、それぞれ対策を定義する。

技術リスク

リスク影響度対策
Firestore単一ドキュメント制限（1MB） 高 CRMデータは個別コレクション方式で回避。詳細は CRMデータモデル設計 を参照
既存 invite_sales への影響 高 読み取り専用参照 + イベントフックのみ。ハイブリッド拡張アーキテクチャの原則に従う
LINE APIの1:N店舗共有パターン 中 user_id → customer_id マッピングテーブルで対応
メールパースの精度 中 manual_review フォールバック + 段階的改善
フロントエンドの複雑化 中 CRMを独立モジュールとして分離（frontend/src/crm/）

運用リスク

リスク影響度対策
既存テナントのマイグレーション 高 lead → customer 自動変換 + 段階的ロールアウト
CRM画面と既存SalesConsoleの共存 中 サイドバーから旧画面にリンク + 段階的移行
Firestoreコスト増加 低 インデックス最適化 + 読み取りキャッシュ

判断が必要な項目

以下は設計書の前提としてユーザーの判断を仰ぎたいポイントをまとめたものである。アーキテクチャ・機能スコープ・UXの3分野に分類する。

アーキテクチャ判断（A1〜A3）

#項目選択肢推奨理由
A1 CRMを同一リポジトリに入れるか 同一 / 別 同一 Firestore・認証・Cloud Runを共有でき開発効率が高い
A2 Firestoreコレクション設計 単一ドキュメント / 個別コレクション 個別コレクション データ量増加に耐えるため
A3 既存SalesConsoleの扱い 維持+並行運用 / CRMに統合して廃止 維持+並行運用 → 段階的にCRMに機能移行 既存テナントへの影響を最小化

機能スコープ判断（S1〜S3）

#項目選択肢推奨理由
S1 MVPでL1を全部やるか 全部 / 段階的 段階的（Phase 1〜3 = 最小MVP） 顧客管理 + AI統合だけで差別化価値が出る
S2 LINE連携の優先度 Phase 4 / もっと後 Phase 4 不動産業界ではLINE必須
S3 メールパースの対象ポータル SUUMO + HOME'S / それ以上 SUUMO + HOME'S のみ シェア上位2つでカバー率80%超

UX判断（U1〜U2）

#項目選択肢推奨理由
U1 CRM主画面 カンバン / タスクリスト 管理者=カンバン、担当者=タスクリスト CRM v3.2 の仕様通り
U2 既存 /sales からCRMへの導線 リダイレクト / ヘッダーリンク ヘッダーリンク 段階的移行で混乱を防ぐ
