フェーズ管理詳細

最終更新: 2026-03-18
Per Heim CRM v3.2における顧客フェーズ（CustomerPhase）の10段階定義、遷移ルール、遷移時の自動アクション、カンバンビューとの対応、フェーズ別KPI設計、および既存lead.statusからのマッピング戦略の詳細仕様。

本記事では、CRMデータモデル設計で概説されたCustomerPhase（顧客フェーズ）の詳細仕様を解説する。Per Heim CRM v3.2 の営業プロセスは、反響受信から成約（または失注）に至るまでの10段階のフェーズで管理され、フロントエンドのカンバンビューと直接対応する。フェーズ遷移は手動操作（カンバンD&D / API呼び出し）と自動トリガー（AIヒアリング完了、カレンダーイベント等）の両方で発生し、遷移時にはタスク自動生成・タイムラインイベント記録・通知が連動する。本記事はPer Heim CRM v3.2 統合設計書のデータモデル設計およびフロントエンド設計を基に、フェーズ管理に特化した詳細仕様を記述する。

CustomerPhaseの定義

顧客の営業進捗は CustomerPhase 型として以下の10値で定義される。各フェーズは不動産仲介業の営業プロセスに対応し、反響受信から成約（または失注）までの顧客ライフサイクルを表現する。

#フェーズ値日本語名説明典型的な滞留期間
1inquiry反響ポータルサイト（SUUMO・HOME'S等）や手動登録による反響受信直後の初期状態。担当者の割当と初回連絡が最優先アクションとなる。0〜1日
2hearing_pendingヒアリング待ちAIヒアリングURL発行済み・未開封の状態。顧客がURLを開封するのを待機している段階。1〜3日
3hearing_activeヒアリング中顧客がAIヒアリングURLを開封し、ヒアリングセッションが進行中の状態。1〜7日
4hearing_doneヒアリング済みAIヒアリングが完了し、プロファイル・ニーズタグが生成された状態。担当者によるヒアリング結果確認と来店提案が次のアクション。1〜5日
5visit_reserved来店予約来店日時が確定し、予約が入っている状態。来店準備（物件資料の用意等）が必要。1〜14日
6visited来店済み来店が完了した状態。ヒアリング結果を基に具体的な物件提案・内見設定を行う段階。1〜7日
7viewing内見物件内見の段階。複数物件の内見を繰り返す場合もこのフェーズに滞留する。7〜30日
8application申込気に入った物件が見つかり、申込手続き中の状態。審査・書類準備が進行する。7〜14日
9contracted成約契約完了の最終状態。KPI集計対象となるゴール地点。（終了状態）
10lost失注何らかの理由で案件が終了した状態。lost_reason フィールドに失注理由を記録する。全フェーズから遷移可能。（終了状態）

型定義としては以下の通りである:

CustomerPhase = Literal[
    "inquiry",           # 反響
    "hearing_pending",   # ヒアリング待ち
    "hearing_active",    # ヒアリング中
    "hearing_done",      # ヒアリング済み
    "visit_reserved",    # 来店予約
    "visited",           # 来店済み
    "viewing",           # 内見
    "application",       # 申込
    "contracted",        # 成約
    "lost",              # 失注
]

フェーズ遷移ルール

フェーズ遷移には許可された遷移パスが定義されており、不正な遷移（例: inquiry から直接 contracted）はAPIレベルで拒否される。ただし、失注（lost）への遷移は全フェーズから許可される。

許可される遷移パス

以下の表は各フェーズから許可される遷移先を示す。順方向遷移（通常の営業フロー）とlost遷移が基本であり、スキップ遷移（例: inquiry → visit_reserved）は原則として許可しない設計である。

hearing_pendinghearing_active, lost顧客のURL開封を自動検知
hearing_activehearing_done, lostセッション完了を自動検知
hearing_donevisit_reserved, lost来店予約の確定で遷移
visit_reservedvisited, lost来店日当日以降に手動確認
visitedviewing, lost内見物件の提案・設定
viewingapplication, lost申込意思の確認
applicationcontracted, lost審査通過・契約完了

lost（遷移なし）最終状態。復活は新規顧客として再登録

自動遷移トリガー

以下のイベントが発生した場合、システムがフェーズを自動的に遷移させる:

トリガーイベント遷移実装箇所
AIヒアリングURL開封（first_opened_at 設定時）hearing_pending → hearing_active既存の session_open エンドポイントにCRMフック追加
AIヒアリングセッション完了（complete_session）hearing_active → hearing_donecomplete_session にCRMイベントフック追加（Phase 3）
Googleカレンダー来店予約作成hearing_done → visit_reservedGoogleカレンダー連携Webhook（Phase 6）

手動遷移

手動でのフェーズ遷移は以下の2つの方法で実行できる:

  • カンバンドラッグ&ドロップ: カンバンビュー上でカードを隣接カラムにドラッグして遷移。内部的には POST /v1/tenant/crm/kanban/move APIを呼び出す。
  • フェーズ遷移API直接呼び出し: POST /v1/tenant/crm/customers/{customer_id}/phase に { phase: CustomerPhase, reason?: string } を送信。理由（reason）はlost遷移時に必須。

いずれの方法でも、遷移の副作用（タイムラインイベント記録・自動タスク生成・通知）が同一のロジックで実行される。遷移ロジックは src/server/crm/phase.py に集約される。

フェーズ遷移時の自動アクション

フェーズ遷移が発生すると、以下の3種類の自動アクションが実行される。これらは遷移の手動/自動を問わず一律に適用される。

タスク自動生成ルール

遷移先フェーズに応じて、担当者の次アクションとなるタスク（auto_generated: true）が自動的に作成される:

遷移先フェーズ自動生成タスクtask_typeprioritydue_date
inquiry初回連絡callhigh当日
hearing_pendingヒアリングURL送付確認follow_upnormal翌営業日
hearing_doneヒアリング結果確認・来店提案follow_uphigh当日
visit_reserved来店準備（物件資料用意）visithigh来店日前日
visited来店フォローアップ・内見提案follow_uphigh来店翌営業日
viewing内見後フォローアップfollow_upnormal内見翌営業日
application申込書類確認documenturgent当日
contracted契約後手続き案内documentnormal3営業日後
lost（タスク生成なし）———

タイムラインイベント記録

全てのフェーズ遷移は event_type: "phase_changed" としてTimelineEventに記録される。記録される情報:

  • summary: 「フェーズ変更: ヒアリング待ち → ヒアリング中」のような可読テキスト
  • detail.from_phase: 遷移元フェーズ
  • detail.to_phase: 遷移先フェーズ
  • detail.reason: 遷移理由（失注時は必須）
  • detail.trigger: "manual" | "auto" | "kanban_dnd"
  • actor_employee_id: 手動遷移時の操作者ID
  • auto_generated: 自動遷移の場合は true

通知トリガー

特定のフェーズ遷移は担当者やマネージャーへの通知を発火する:

  • hearing_done到達: 担当者に「ヒアリング完了 — 結果を確認してください」通知
  • visit_reserved到達: 担当者に「来店予約が確定しました」通知
  • application到達: マネージャーに「申込案件が発生しました」通知
  • contracted到達: マネージャー＋管理者に「成約しました」通知
  • lost到達: マネージャーに「失注: {顧客名} — 理由: {lost_reason}」通知

カンバンビューとの対応

フロントエンド設計で定義されたカンバンビュー（/crm/kanban）は、CustomerPhaseの各値をカラムとして表示する。カンバンUIコンポーネントは KanbanBoard.tsx および KanbanColumn.tsx で実装される。

カラムとフェーズの対応

カンバンボードにはlostとcontractedを除く8つのアクティブフェーズがカラムとして表示される。成約・失注は別セクション（完了済み一覧）で確認する。

表示順カラム名対応フェーズカラム背景色
1反響inquiry薄い赤
2ヒアリング待ちhearing_pending薄いオレンジ
3ヒアリング中hearing_active薄い黄
4ヒアリング済みhearing_done薄い黄緑
5来店予約visit_reserved薄い緑
6来店済みvisited薄い青緑
7内見viewing薄い青
8申込application薄い紫

カラム内ソート基準

各カラム内のカード（顧客）は以下の優先順でソートされる:

  1. フェーズ滞留日数（降順）: 長期滞留している顧客が上に表示され、対応漏れを防止する
  2. タスク期限（昇順）: 未完了タスクのうち最も期限が近いものを基準にソート
  3. 更新日時（降順）: 直近にアクションのあった顧客を上位表示

カードに表示される情報

各カンバンカード（CustomerSummary）には以下が表示される:

  • 顧客名
  • 流入元（SUUMO / HOME'S / 手動 等のアイコン）
  • フェーズ滞留日数（例: 「3日前」）
  • 担当者名
  • ヒアリング完了率（hearing系フェーズの場合）
  • 来店予約日時（visit_reservedフェーズの場合）

ドラッグ&ドロップ操作

カードのドラッグ&ドロップは隣接カラムへの移動のみ許可される。2つ以上離れたカラムへの直接移動や、逆方向への移動はUIレベルで制限される。D&D操作は内部的に POST /v1/tenant/crm/kanban/move API を呼び出し、フェーズ遷移の全副作用（タイムラインイベント・タスク生成・通知）が実行される。カンバンカラム幅は --crm-kanban-column-width: 280px で定義される。

フェーズ別KPI設計

フェーズ管理データは統合設計書の分析・レポート設計と連動し、営業パフォーマンスの定量評価に活用される。

リードタイム計測

各フェーズ間の遷移にかかった日数をリードタイムとして計測する。phase_changed_at の差分から算出し、GET /v1/tenant/crm/analytics/lead-time APIで取得できる。

遷移区間KPI名目標値（参考）計測方法
inquiry → hearing_pending初回対応時間1時間以内反響受信〜URL発行までの経過時間
hearing_pending → hearing_doneヒアリング完了期間3日以内URL発行〜セッション完了
hearing_done → visit_reserved来店予約獲得期間5日以内ヒアリング完了〜来店予約確定
visit_reserved → visited来店実現期間14日以内予約〜来店完了
visited → contracted成約期間30日以内来店〜契約完了
inquiry → contracted全体リードタイム60日以内反響受信〜契約完了

コンバージョン率

各フェーズ間の遷移率をコンバージョン率として追跡する。ファネル分析APIは GET /v1/tenant/crm/analytics/funnel で提供される。

コンバージョン指標計算式意味
ヒアリング到達率hearing_done数 / inquiry数反響からヒアリング完了に至った割合
来店予約率visit_reserved数 / hearing_done数ヒアリング完了から来店予約に至った割合
来店実現率visited数 / visit_reserved数予約した顧客が実際に来店した割合
成約率（来店後）contracted数 / visited数来店から成約に至った割合
全体成約率contracted数 / inquiry数反響から成約に至った全体割合
失注率lost数 / inquiry数失注した全体割合

デュアルファネルとの関係

Per Heim CRMの分析設計ではデュアルファネルを採用する。これはPer Heim AIヒアリングを経由した顧客と、それ以外の顧客を分離してコンバージョン率を比較するものである。ファネルAPIの compare_perheim パラメータで切り替え可能。

  • 全体ファネル（overall_funnel）: 全反響からのコンバージョン率
  • Per Heimファネル（perheim_funnel）: AIヒアリングを完了した顧客のみのコンバージョン率

この比較により、AIヒアリング導入のROI（投資対効果）を定量的に評価できる。Per Heimを経由した顧客は、ニーズの可視化により来店率・成約率が向上することが期待される。

lead.statusからのマッピング

既存の lead データ（招待単位のレコード）を新しい Customer モデルに移行する際、lead.status の5つの値を CustomerPhase にマッピングする必要がある。詳細なマイグレーション戦略はCRMデータモデル設計のマイグレーションセクションを参照。

変換テーブル

lead.statuscustomer.phaseマッピング根拠
newhearing_pending新規作成されたlead = URL発行済み・未開封状態に相当。既存leadは手動URL発行後に作成されるため、inquiry（反響受信直後）ではなくhearing_pendingにマッピング。
openedhearing_activeURL開封済み = ヒアリングセッション進行中に相当。first_opened_at が設定されている状態。
in_progresshearing_active進行中もヒアリング実行中の状態に相当。opened と in_progress はCRM上では同一フェーズとして扱う。
completedhearing_doneヒアリング完了状態に直接対応。プロファイル・ニーズタグが生成済み。
revokedlost取消（管理者による無効化）= 失注に相当。lost_reason に「マイグレーション: lead取消」を記録。

点線で示したフェーズ（inquiry, visit_reserved〜contracted）は、既存leadからのマッピング対象外であり、CRM運用開始後に新規に発生するフェーズである。既存データのマイグレーションではAIヒアリングの段階（hearing系3フェーズ + lost）にのみマッピングされる。

マッピング実装

マイグレーション関数の核心部分は以下の通りである:

phase_map = {
    "new": "hearing_pending",
    "opened": "hearing_active",
    "in_progress": "hearing_active",
    "completed": "hearing_done",
    "revoked": "lost",
}

customer.phase = phase_map.get(lead["status"], "inquiry")

マッピングに該当しない未知のstatus値はフォールバックとして inquiry に設定される。初回マイグレーションは1 lead = 1 customerの暫定方式で実施し、同一人物の統合（merge）はCRMデータモデル設計で説明されているmerge UIおよび重複検知（Phase 5）の導入後に段階的に実施する。
