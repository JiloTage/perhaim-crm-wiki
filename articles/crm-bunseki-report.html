分析レポート詳細

最終更新: 2026-03-18
Per Heim CRM v3.2における分析・レポート機能の詳細設計。デュアルファネルによる全体/Per Heim経由の比較分析、ROI訴求指標（RoiMetrics）による効果測定、ダッシュボードKPI、リードタイム分析、月次レポートの自動生成、およびMVPから将来拡張に向けた集計アーキテクチャの移行計画を定義する。Phase 6（分析・レポート）の実装仕様に対応する。

デュアルファネル

CRM v3.2の分析機能の中核は、全体ファネルとPer Heim経由ファネルの2系統を並列比較するデュアルファネル設計にある。この設計により「Per Heimを使うと来店率がX%上がる」ことをデータで証明でき、経営者へのROI訴求力を最大化する。

ファネルステージ定義

各ファネルは以下の4ステージで構成される。全体ファネルとPer Heim経由ファネルでステージ名称が若干異なる点に注意する。

ステージ全体ファネルPer Heim経由ファネル定義
Stage 1 反響 ヒアリング依頼 顧客からの初回接触。全体は全チャネル（SUUMO、HOME'S、直接問合せ等）、Per Heim経由はAIヒアリング依頼を起点とする
Stage 2 ヒアリング中 ヒアリング完了 顧客ニーズの把握フェーズ。Per Heim経由はAIヒアリング完了（自動遷移）を基準とする
Stage 3 来店予約 来店予約 物件見学または相談のための来店予約。来店率の算出基準となるステージ
Stage 4 成約 成約 契約締結。最終コンバージョン

ファネル比較例

ステージ全体ファネルPer Heim経由ファネル
反響 / ヒアリング依頼45件20件
ヒアリング中 / 完了20件15件
来店予約8件（来店率 17.8%）7件（来店率 35.0%）
成約3件（成約率 6.7%）3件（成約率 15.0%）

Per Heim経由の来店率は全体の約2倍（35.0% vs 17.8%）であり、AIヒアリングによる顧客ニーズの事前把握が来店意欲を大幅に向上させることを定量的に示している。

可視化仕様

ファネルデータは CRM API設計 の分析APIエンドポイント GET /v1/tenant/crm/analytics/funnel から取得する。クエリパラメータ compare_perheim=true を指定することで、2系統のファネルデータが同時に返却される。

  • 表示形式: 2カラムの逆三角形ファネルチャート（左: 全体、右: Per Heim経由）
  • 色分け: 全体ファネルは青系グラデーション、Per Heim経由は緑系グラデーション
  • 遷移率: 各ステージ間の遷移率をパーセンテージで表示
  • 期間フィルタ: 7日 / 30日 / 90日 の切替に対応

ROI訴求指標（RoiMetrics）

RoiMetricsモデルは、Per Heim経由と非Per Heim経由の顧客行動を定量的に比較し、AIヒアリングの投資対効果を経営者に訴求するための指標セットである。月次ベースで集計し、改善トレンドの追跡を可能にする。

Per Heim経由 vs 非経由の比較指標

指標カテゴリPer Heim経由フィールド非Per Heimフィールド説明
反響数 perheim_inquiry_count non_perheim_inquiry_count 当該期間の新規反響件数
来店率 perheim_visit_rate non_perheim_visit_rate 反響→来店予約の遷移率（%）
成約率 perheim_contract_rate non_perheim_contract_rate 反響→成約の遷移率（%）
平均リードタイム perheim_avg_lead_time_days non_perheim_avg_lead_time_days 反響から成約までの平均日数

RoiMetricsモデル定義

RoiMetricsはPydantic BaseModelとして定義され、以下の差分指標フィールドを自動算出する。

差分フィールド型算出ロジック意味
visit_rate_improvement float perheim_visit_rate - non_perheim_visit_rate 来店率改善ポイント（例: +17.2pt）
contract_rate_improvement float perheim_contract_rate - non_perheim_contract_rate 成約率改善ポイント
lead_time_reduction_days float non_perheim_avg_lead_time_days - perheim_avg_lead_time_days リードタイム短縮日数（正値=Per Heimが速い）

periodフィールド（例: "2026-03"）により月次単位での比較データを管理する。過去データとの比較により、Per Heimの効果が時間とともにどう推移するかのトレンド分析も可能となる。

ダッシュボードKPI

CRMダッシュボードのトップには、営業チームが日々確認すべき6つのKPI指標を配置する。データは GET /v1/tenant/crm/analytics/dashboard エンドポイントから取得し、7日 / 30日 / 90日の期間切替に対応する。

KPIフィールド名型定義用途
総顧客数 total_customers int CRM内の全顧客レコード数（アクティブ+失注含む） CRM全体の規模把握
新規反響数 new_inquiries int 指定期間内に受信した新規反響件数 マーケティング効果の指標
ヒアリング完了率 hearing_completion_rate float ヒアリング依頼のうち完了した割合（%） AIヒアリングの稼働状況
来店率 visit_rate float 反響から来店予約に至った割合（%） 追客効果の指標
成約率 contract_rate float 反響から成約に至った割合（%） 営業パフォーマンスの指標
平均リードタイム avg_lead_time_days dict[str, float] フェーズ間遷移の平均日数（辞書形式で各遷移ペアを保持） 営業プロセスの速度指標

各KPIカードには前期比の増減（矢印アイコン+増減率）を併記し、改善/悪化のトレンドを一目で把握できるようにする。

リードタイム分析

リードタイム分析は、顧客が各フェーズ間を遷移するのに要した平均日数を算出し、営業プロセスのボトルネックを特定するための機能である。データは GET /v1/tenant/crm/analytics/lead-time から取得する。

平均日数の算出

レスポンスは phase_transitions 配列として返却され、各要素は以下の構造を持つ。

フィールド型説明
from_phasestr遷移元フェーズ（例: "inquiry"）
to_phasestr遷移先フェーズ（例: "hearing_done"）
avg_daysfloat遷移に要した平均日数

ボトルネック検出

リードタイムデータをもとに、以下のルールでボトルネックを自動検出する。

  • 閾値超過: 特定のフェーズ遷移の平均日数が業界ベンチマーク（例: 反響→来店で14日以上）を超えた場合にアラート表示
  • 前月比悪化: 前月と比較してリードタイムが20%以上延長した遷移をハイライト
  • Per Heim比較: Per Heim経由のリードタイムと非経由のリードタイムを比較し、AIヒアリングによる短縮効果を可視化

改善提案

ボトルネックが検出されたフェーズ遷移に対して、システムは以下の改善提案を自動表示する。

  • 反響→ヒアリングが長い場合: 「初回連絡の自動化（サンクスメール+LINE通知）を検討してください」
  • ヒアリング→来店が長い場合: 「来店促進テンプレートの活用、またはPer Heimヒアリングの導入を検討してください」
  • 来店→成約が長い場合: 「追客コミュニケーションの頻度を見直してください」

月次レポート

月次レポートは、毎月1日にCloud Schedulerがトリガーする自動生成レポートである。経営者向けのサマリーと現場向けの詳細データの両方を含む構成とする。データは GET /v1/tenant/crm/analytics/monthly-report?month=YYYYMM から取得可能。

レポート構成要素

セクション内容対象
エグゼクティブサマリー 月次KPI概要、前月比、目標達成率 経営者
デュアルファネル 全体 vs Per Heim経由の月次ファネル比較 経営者
ROI指標 RoiMetricsの月次推移、Per Heim効果の定量データ 経営者
営業パフォーマンス 担当者別成約数、対応速度、タスク完了率 マネージャー
リードタイム分析 フェーズ別平均日数、ボトルネック、改善提案 マネージャー
反響チャネル分析 SUUMO/HOME'S/直接問合せの比率、チャネル別成約率 マーケティング

出力仕様

  • PDF出力: 経営者・マネージャー向け。ファネルチャート、KPIグラフをビジュアルに含む。Cloud Run上でレンダリングし、Firestore Storageに保存
  • CSV出力: データ分析向け。全KPIの生データ、顧客別フェーズ遷移履歴、担当者別パフォーマンスデータをエクスポート可能
  • 配信: 毎月1日の monthly_report Cloud Schedulerジョブにより自動生成し、管理者・マネージャーにメールで配信

集計アーキテクチャ

分析データの集計方式は、MVP段階のリアルタイム集計から将来拡張のバッチ集計へ段階的に移行する設計とする。データ量が増加してもAPI応答時間200ms以内の非機能要件を維持するための戦略である。

レビュー指摘 L2 対応: リアルタイム集計（count on read）はデータ量増加で早期にボトルネックとなる。カウンター方式を推奨する: フェーズ遷移時にインクリメンタルにカウンター更新し、crm_analytics/{tenant_id}/counters/{counter_type} でカウンターを管理する。日次/月次の正確な集計はバッチジョブで補正・更新する。

MVP段階: リアルタイム集計

MVP段階ではテナントあたりの顧客数が少ない（想定: 数百件以下）ため、Firestoreクエリで毎回算出する方式を採用する。

  • 集計方式: APIリクエスト時にFirestoreの crm_customers / crm_timeline コレクションに対してクエリを実行
  • メリット: 実装コストが低い、データの即時反映、バッチ処理の運用不要
  • 制約: 顧客数1,000件超でレスポンスタイム200ms超過のリスク
  • Firestoreインデックス: CRM実装フェーズ計画で定義した複合インデックスを活用

将来拡張: バッチ集計

顧客数の増加に伴い、crm_analytics/{tenant_id}/monthly/{yyyymm} コレクションに日次バッチで集計済みデータを保存する方式へ移行する。

  • 集計タイミング: Cloud Schedulerが毎日深夜（03:00 JST）に集計ジョブをトリガー
  • 保存先: crm_analytics/{tenant_id}/monthly/{yyyymm} ドキュメントにKPI・ファネル・リードタイム等の集計済み値を保存
  • API側の変更: 集計済みデータが存在する場合はそちらを参照し、なければリアルタイム集計にフォールバック
  • 移行トリガー: テナントの顧客数が1,000件を超えた時点で自動的にバッチ集計モードに切り替え

移行計画

段階条件集計方式API応答時間（目安）
MVP 顧客数 < 1,000件 Firestoreリアルタイムクエリ 100〜200ms
中間 顧客数 1,000〜10,000件 日次バッチ集計 + リアルタイムフォールバック 30〜50ms（集計済み）/ 200ms+（フォールバック）
大規模 顧客数 > 10,000件 完全バッチ集計 + キャッシュ < 30ms

集計アーキテクチャ全体は Per Heim CRM v3.2 統合設計書 の非機能要件（パフォーマンス基準: API応答200ms以内）を満たすように設計されている。Phase 6の実装スコープと依存関係については CRM実装フェーズ計画 を参照。
