カンバン実装詳細

最終更新: 2026-03-18
Per Heim CRM v3.2 のカンバンビュー（/crm/kanban）の実装設計を詳述する。KanbanBoard・KanbanColumn・CustomerCard の3層コンポーネント構成、@dnd-kit によるドラッグ&ドロップ、楽観的UI更新とロールバック戦略、8カラムのフェーズ別表示仕様、パフォーマンス最適化手法を包括的に定義する。

Per Heim CRM v3.2 のカンバンビューは、CustomerPhase（顧客フェーズ）をカラムとして顧客カードを視覚的に管理する中核画面である。 営業担当者はドラッグ&ドロップ操作でカード（顧客）をカラム間で移動させ、フェーズ遷移をシームレスに実行できる。 フロントエンド全体の設計方針はCRMフロントエンド設計を、 フェーズ遷移のビジネスロジックはCRMフェーズ管理を、 バックエンドAPIの仕様はCRM API設計を参照のこと。

カンバンコンポーネント設計

カンバンビューは3層のコンポーネント階層で構成される。各コンポーネントの責務を明確に分離し、再利用性とテスタビリティを確保する。

KanbanBoard（ボード層）

KanbanBoard.tsx はカンバン全体のコンテナコンポーネントである。以下の責務を持つ:

  • 横スクロール可能なフレックスコンテナの提供（overflow-x: auto）
  • @dnd-kit の DndContext プロバイダーとしてドラッグ&ドロップのスコープを定義
  • useCustomers hooks を通じたカンバンAPI（GET /v1/tenant/crm/kanban）からのデータ取得
  • フィルタ条件（store_id、assigned_to）のクエリパラメータ管理

Props / State 設計:

// KanbanBoard Props
interface KanbanBoardProps {
  storeId?: string;       // 店舗フィルタ
  assignedTo?: string;    // 担当者フィルタ
}

// KanbanBoard 内部 State
const [columns, setColumns] = useState<KanbanColumnData[]>([]);
const [activeCard, setActiveCard] = useState<CustomerSummary | null>(null);
const [isMoving, setIsMoving] = useState(false);

KanbanColumn（カラム層）

KanbanColumn.tsx は個別のフェーズカラムを描画するコンポーネントである:

  • カラムヘッダー（フェーズ名 + 件数バッジ）の表示
  • @dnd-kit の useDroppable によるドロップ領域の定義
  • カラム内の CustomerCard リストの描画とソート制御
  • カラム幅は CSS変数 --crm-kanban-column-width: 280px で統一

Props 設計:

interface KanbanColumnProps {
  phase: CustomerPhase;
  title: string;
  customers: CustomerSummary[];
  count: number;
  colorToken: string;  // --crm-phase-inquiry 等のCSS変数名
}

ドラッグ&ドロップ実装

カンバンビューのドラッグ&ドロップは、React向けD&Dライブラリ @dnd-kit を採用して実装する。

ライブラリ選定

D&Dライブラリの選定にあたり、主要2ライブラリを比較した結果 @dnd-kit を採用する:

項目@dnd-kitreact-beautiful-dnd
メンテナンス状況活発（2024年〜）Atlassianがメンテナンス終了（非推奨）
React 18対応完全対応StrictMode で問題あり
アクセシビリティ組み込みサポート（キーボード操作・スクリーンリーダー）基本サポート
カスタマイズ性hooks ベースで高い柔軟性限定的
バンドルサイズ約 12KB (gzip)約 30KB (gzip)
タッチデバイスPointerSensor で統合対応別途設定必要

採用理由: react-beautiful-dnd はAtlassianによるメンテナンスが終了しており、React 18 の StrictMode との互換性問題がある。@dnd-kit はhooksベースのモダンなAPIを提供し、バンドルサイズが小さく、アクセシビリティサポートも充実している。

ドラッグイベントハンドリング

KanbanBoard 内の DndContext で以下の3イベントをハンドリングする:

// KanbanBoard.tsx 内
<DndContext
  sensors={sensors}
  collisionDetection={closestCorners}
  onDragStart={handleDragStart}
  onDragOver={handleDragOver}
  onDragEnd={handleDragEnd}
>
  {columns.map(col => (
    <KanbanColumn key={col.phase} {...col} />
  ))}
  <DragOverlay>
    {activeCard && <CustomerCard customer={activeCard} isDragging />}
  </DragOverlay>
</DndContext>
イベントトリガー処理内容
onDragStartカードをつかんだ瞬間activeCard state にドラッグ中の顧客情報をセット。DragOverlay にゴーストカードを表示
onDragOver別カラム上にカードが通過ドロップ先カラムのハイライト表示。ただしフェーズ遷移ルールに反する場合は 禁止アイコン を表示
onDragEndカードをドロップ楽観的UI更新を実行し、POST /v1/tenant/crm/kanban/move APIを呼び出す（後述）

ドロップ時のフェーズ遷移API呼び出し

カードがドロップされた時点で、以下のフローが実行される:

  1. ドロップ先カラムの phase を取得
  2. 同一カラムへのドロップの場合は何もしない（カラム内ソート変更は将来対応）
  3. 楽観的UI更新を即座に適用（後述）
  4. カンバンAPI の POST /v1/tenant/crm/kanban/move を呼び出し: 
    POST /v1/tenant/crm/kanban/move
Body: {
  customer_id: string,
  to_phase: CustomerPhase,
  reason?: string    // オプション：遷移理由モーダルで入力
}
  5. API応答に基づき確定またはロールバック

楽観的UI更新

カンバンビューでは楽観的UI更新（Optimistic UI Update）パターンを採用し、ドラッグ操作の体感速度を最大化する。ユーザーがカードをドロップした瞬間にUIを即座に更新し、バックグラウンドでAPIリクエストを送信する。

楽観的更新フロー

ロールバック戦略

スナップショット方式を採用し、ドラッグ操作前のカラムデータ全体を保持する:

const handleDragEnd = async (event: DragEndEvent) => {
  const { active, over } = event;
  if (!over) return;

  const fromPhase = active.data.current?.phase;
  const toPhase = over.id as CustomerPhase;
  if (fromPhase === toPhase) return;

  // 1. スナップショット保存
  const snapshot = structuredClone(columns);

  // 2. 楽観的更新（即座にUI反映）
  setColumns(prev => moveCardBetweenColumns(prev, active.id, fromPhase, toPhase));

  try {
    // 3. API呼び出し
    await moveCustomerPhase(active.id as string, toPhase, reason);
    // 4a. 成功 → 確定
    toast.success(`${customerName} を ${phaseLabel(toPhase)} に移動しました`);
  } catch (error) {
    // 4b. 失敗 → ロールバック
    setColumns(snapshot);
    toast.error(getErrorMessage(error));
  }
};

ロールバックが発生するケース:

  • フェーズ遷移ルール違反: 例えば inquiry → contracted への直接遷移など、フェーズ管理で定義されたルールに反する移動（APIが400を返す）
  • 権限不足: 担当者以外が他担当の顧客を遷移させようとした場合
  • ネットワークエラー: API通信が失敗した場合
  • 同時操作の競合: 他ユーザーが同一顧客を同時に操作した場合（409 Conflict）

遷移理由モーダル

特定のフェーズ遷移（例: lost への遷移）時には、ドロップ後にオプションの理由入力モーダルを表示する。モーダルで「キャンセル」が選択された場合はロールバックを実行する。

カンバンカラム仕様

カンバンビューは CustomerPhase の10フェーズのうち8カラムを表示する。

表示カラム一覧

#カラム名フェーズ値背景色トークン説明

2ヒアリング待ちhearing_pending--crm-phase-hearingヒアリングURL発行済み・回答待ち
3ヒアリング中hearing_active--crm-phase-hearingAIヒアリング回答中
4ヒアリング済みhearing_done--crm-phase-hearingヒアリング完了・条件整理済み

除外フェーズ: lost / contracted

以下の2フェーズはカンバンボードにカラムとして表示しない:

  • lost（失注）: 営業パイプラインから離脱した顧客であり、カンバンで視覚的に管理する対象ではない。失注顧客は顧客一覧（/crm/customers）でフィルタ表示する
  • contracted（成約）: 営業プロセスが完了した顧客であり、パイプライン管理の対象外。成約実績はダッシュボードのKPIカードや分析画面で確認する

ただし、カンバン上のカードを lost にドラッグする操作は、ボード下部に表示される「失注にする」ドロップゾーンで対応する。この操作時には失注理由の入力モーダルが必須で表示される。

カラム内ソート

各カラム内の顧客カードは以下の3段階優先度でソートされる:

  1. 優先度（priority）: high → medium → low（タスクの緊急度に基づく）
  2. 期限（next_action_deadline）: 期限が近い順（昇順）。期限切れの顧客が最上部に表示される
  3. 更新日時（updated_at）: 最終更新が古い順（昇順）。長期間放置されている顧客が上位に来る

このソートロジックにより、対応が急務な顧客カードがカラム上部に自然と集まる設計となっている。

CustomerCardコンポーネント

CustomerCard.tsx はカンバン上の個別顧客カードの表示を担当する。コンパクトなカード形式で必要最小限の情報を表示し、詳細は顧客カード画面（/crm/customers/:id）に委ねる。

カード内表示情報

表示項目データソース表示例
顧客名customer.name田中太郎
ソースアイコンcustomer.sourceSUUMO / HOME'S / 手動 等のアイコン
経過時間customer.phase_changed_at からの経過5分前 / 2時間前 / 3日前
担当者customer.assigned_to_name担当: 営業A

カードの視覚的要素:

  • カード左端にフェーズカラー（CSS変数 --crm-phase-*）のアクセントボーダーを表示
  • 経過時間が 24時間超 の場合、カード背景に薄い警告色を適用（対応遅延の視覚的アラート）
  • ドラッグ中はカードに影（box-shadow）を追加し、元の位置に半透明のプレースホルダーを表示

クリック時の顧客詳細遷移

カードのクリック（ドラッグ開始と判定されない短いタップ）で /crm/customers/:customerId へ navigate() で遷移する。@dnd-kit の PointerSensor に activationConstraint（距離: 8px）を設定することで、クリックとドラッグを明確に区別する:

const sensors = useSensors(
  useSensor(PointerSensor, {
    activationConstraint: {
      distance: 8,  // 8px 以上の移動でドラッグ開始
    },
  }),
  useSensor(KeyboardSensor, {
    coordinateGetter: sortableKeyboardCoordinates,
  })
);

パフォーマンス最適化

カンバンビューは数百件の顧客カードを同時に表示する可能性があるため、以下のパフォーマンス最適化を実装する。

仮想スクロール

各カラム内のカードリストに react-window（または @tanstack/react-virtual）による仮想スクロールを適用する。表示領域に入ったカードのみDOMにレンダリングし、100件以上のカードがあるカラムでも60fpsを維持する。

  • カード高さは固定（estimateSize: 120px）として計算を高速化
  • オーバースキャン（前後5件）で高速スクロール時の空白を防止
  • ドラッグ中は仮想化を一時的に解除し、ドロップ先の正確な位置計算を保証

メモ化（React.memo）

不要な再レンダリングを防止するため、以下のメモ化戦略を適用する:

  • KanbanColumn を React.memo でラップし、customers 配列の参照が変わらない限り再レンダリングしない
  • CustomerCard を React.memo でラップし、customer オブジェクトの updated_at が変わらない限り再レンダリングしない
  • ドラッグ&ドロップのイベントハンドラは useCallback でメモ化
  • カラムデータのフィルタ・ソート処理は useMemo で計算結果をキャッシュ

リアルタイム更新（Firestore onSnapshot）

カンバンデータはFirestoreの onSnapshot リスナーによるリアルタイム同期を行う。複数の営業担当者が同時にカンバンを操作する場面を想定し、以下の設計とする:

  • crm_customers コレクションに対して onSnapshot リスナーを設定し、他ユーザーによるフェーズ変更を即座に反映
  • 自身の楽観的更新と外部からの更新が競合した場合は、サーバー側のデータを優先（last-write-wins）
  • 更新頻度が高い場合は debounce（300ms） を適用し、UIのちらつきを防止
  • バックグラウンドタブではリスナーを pause し、フォアグラウンド復帰時に再同期

大量データ対応

顧客数が増加した場合の対策:

  • ページネーション: 各カラムの初期表示は最大50件とし、「さらに読み込む」ボタンで追加取得
  • フィルタの活用: 店舗・担当者フィルタで表示対象を絞り込み、初期ロード量を削減
  • バックグラウンドプリフェッチ: 初期表示後に残りのカラムデータを非同期で取得
  • Firestoreクエリ最適化: where + orderBy + limit を組み合わせた複合インデックスの活用
