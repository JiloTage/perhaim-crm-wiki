CRM外部連携詳細

最終更新: 2026-03-18
Per Heim CRM v3.2 における外部連携設計の詳細仕様。LINE Messaging API、Gmail反響取込、Google Calendar、CSVインポートの4つの連携チャネルについて、アーキテクチャ・データフロー・エラーハンドリングを網羅的に定義する。すべての連携モジュールは src/server/integrations/ に集約され、CRMビジネスロジックとは疎結合に保たれる。

LINE Messaging API連携

Per Heim CRMはLINE公式アカウントと連携し、顧客とのメッセージングを一元管理する。テナント設定画面でLINE Channel Access TokenおよびChannel Secretを登録し、Webhook URLを通じてリアルタイムにメッセージを受信する。

Webhook受信

LINE PlatformからのWebhookは /v1/webhook/line-message?tenant_id=xxx エンドポイントで受信する。受信時の処理フローは以下の通り:

  1. Channel Secretを用いた署名検証（X-Line-Signature ヘッダー）
  2. line_user_id から customer_id へのマッピング解決
  3. メッセージを crm_communications コレクションに記録
  4. crm_timeline にイベントとして記録
  5. 担当者へのリアルタイム通知（Web Push + バッジ）

Reply/Push API送信

CRM画面からの返信は2種類のAPIを使い分ける:

API用途制約
Reply API受信メッセージへの即時返信replyToken有効期限内（約30秒）のみ使用可能
Push API任意タイミングでのメッセージ送信月間送信数に応じた従量課金

CRM画面で担当者が返信ボタンを押下した場合、replyTokenが有効であればReply APIを優先し、期限切れの場合はPush APIにフォールバックする。

line_user_id → customer_id マッピング

line_user_id はLINEプラットフォームが発行するユーザー固有IDである。CRM側では crm_customers ドキュメントの line_user_id フィールドでマッピングを管理する。初回メッセージ受信時にマッピングが存在しない場合は、新規顧客レコードを自動作成し、フェーズを new に設定する。

1:N 店舗共有パターン

不動産業界では1つのLINE公式アカウントを複数店舗で共有するケースが一般的である。この場合の顧客振り分けロジック:

  • 初回メッセージ: テナント設定のデフォルト店舗、またはラウンドロビンで割当
  • 既存顧客: customer_id に紐づく store_id の担当者にルーティング
  • 店舗間移動: manager以上の権限で store_id を変更可能

Gmail反響取込

SUUMO、HOME'S、at home などの不動産ポータルサイトからの反響メールを専用Gmailアドレスに転送し、Gmail API経由で自動取得・パースする。パースされた情報は crm_inquiries コレクションに格納され、CRMの反響受信トレイに表示される。

メール転送パース戦略

各ポータルサイトのメールフォーマットに対応したパターンマッチングパーサーを実装する:

ポータルパース対象識別方法
SUUMO顧客名、電話番号、メール、問い合わせ物件、メッセージ送信元ドメイン + 件名パターン
HOME'S顧客名、電話番号、メール、問い合わせ物件、メッセージ送信元ドメイン + HTMLテンプレート構造
at home顧客名、電話番号、メール、問い合わせ物件、メッセージ送信元ドメイン + テキストフォーマット

パーサーは integrations/gmail.py 内にポータル別のクラスとして実装し、新規ポータル追加時は新しいパーサークラスを追加するだけで対応可能なプラグイン構造とする。

Gmail API Polling vs Push Notification

方式実装メリットデメリット
A: Gmail API PollingCloud Scheduler → Cloud Run → Gmail APIシンプル実装、運用容易最大数分の遅延（ポーリング間隔依存）
B: Gmail Push NotificationGmail → Pub/Sub → Cloud Runリアルタイム受信Pub/Sub設定が複雑、認証管理コスト増

MVP判断: 方式A（Polling、1分間隔）を採用する。不動産反響は数分の遅延が許容される業務であり、実装コストと運用負荷を最小化する。リアルタイム性が必要になった段階でPub/Sub連携に移行可能。Cloud Schedulerのジョブは /v1/internal/poll-gmail エンドポイントを定期呼び出しする。

パース失敗時のmanual_reviewフォールバック

ポータルサイトのメールフォーマット変更や未対応ポータルからのメール受信時、パースが失敗する可能性がある。この場合:

  1. パース例外をキャッチし、status: "manual_review" で crm_inquiries に保存
  2. 元メールの件名・本文・送信元をそのまま格納（raw_email フィールド）
  3. CRM反響受信トレイで「要手動確認」バッジを表示
  4. 担当者が手動で顧客情報を入力し、顧客レコードにリンク

Google Calendar連携

CRM画面での来店予約登録をGoogle Calendar APIと双方向同期する。カレンダーイベントの変更をWebhook通知で検知し、顧客フェーズの自動遷移を実現する。

イベントCRUD

CRM画面から以下の操作をGoogle Calendar APIに同期する:

CRM操作Calendar API備考
来店予約登録events.insert()顧客名・物件名をイベントタイトルに含める
予約日時変更events.update()CRM側のタイムラインにも変更記録
予約キャンセルevents.delete()フェーズを visit_reserved から前フェーズに戻す

Google CalendarのイベントIDは crm_customers ドキュメントの calendar_event_ids 配列で管理する。1顧客に対して複数回の来店予約が可能なため、配列構造を採用する。

Webhook通知によるフェーズ自動遷移

Google Calendar APIのWebhook (Push Notification)を利用し、カレンダーイベントの変更をリアルタイムに検知する。Webhook URL は /v1/webhook/google-calendar?tenant_id=xxx とする。

主要なフェーズ自動遷移パターン:

トリガー遷移前フェーズ遷移後フェーズ条件
来店予約作成approach / respondingvisit_reservedCRM経由でイベント作成時
来店完了（イベント時刻経過）visit_reservedvisitedイベント終了時刻 + バッファ時間経過
予約キャンセルvisit_reserved前フェーズに復帰イベント削除検知時

CSVインポート

既存顧客データや他システムからのデータ移行のため、CSVファイルによる一括インポート機能を提供する。フロントエンドとバックエンドの連携による10ステップフローで、安全かつ確実なデータ取り込みを実現する。

フロントエンド/バックエンドの10ステップフロー

ステップ実行側処理内容
1フロントエンドファイル選択 — CSVファイルをドラッグ&ドロップまたはファイルダイアログで選択
2フロントエンドプレビュー表示 — 先頭10行を表形式でプレビュー、文字コード自動判定（UTF-8/Shift_JIS）
3バックエンドパース — CSVファイルをアップロード、ヘッダー行とデータ行を分離
4フロントエンドカラムマッピングUI — CSVカラムとCRMフィールドの対応をドラッグ&ドロップで設定
5バックエンドバリデーション — 必須フィールド、データ型、フォーマット（電話番号・メール等）検証
6バックエンド重複検知 — 電話番号・メールアドレスによる既存顧客との重複チェック
7フロントエンドドライラン確認 — バリデーション結果と重複候補を一覧表示、ユーザー確認
8バックエンドドライラン結果 — 新規作成/更新/スキップの件数サマリーを返却
9フロントエンドインポート実行 — ユーザーが最終確認ボタンを押下
10バックエンドBulk Insert — Firestoreバッチ書き込み（500件/バッチ上限）、結果レポート返却

カラムマッピング

CRMの顧客フィールドとCSVカラムのマッピングは、以下の方式で柔軟に対応する:

  • 自動マッピング: ヘッダー名の類似度判定（「氏名」→ name、「電話」→ phone 等）
  • 手動マッピング: ドラッグ&ドロップUIで任意のカラム対応を設定
  • マッピングテンプレート: よく使うマッピングパターンを保存・再利用

ドライラン・バリデーション

本番インポート前に必ずドライランを実行し、以下を検証する:

  • 必須フィールド検証: name は必須、phone または email のいずれかが必須
  • フォーマット検証: 電話番号（正規表現）、メールアドレス（RFC準拠）、日付フォーマット
  • 重複検知: 電話番号・メールアドレスの完全一致で既存顧客を検索
  • 結果サマリー: 新規作成N件 / 更新N件 / エラーN件 / スキップN件

連携基盤設計

4つの外部連携モジュールはすべて src/server/integrations/ ディレクトリに集約される。各モジュールは共通のエラーハンドリングパターンとリトライ戦略に従い、CRMビジネスロジック層（src/server/crm/）とは明確に分離される。

integrations/ ディレクトリ構造

ファイル役割連携先
integrations/__init__.pyパッケージ初期化、共通インターフェース定義-
integrations/line.pyLINE Messaging API — Webhook受信、Reply/Push送信LINE Platform
integrations/gmail.pyGmail API — 反響メール取込、ポータル別パーサーGmail API
integrations/google_calendar.pyGoogle Calendar API — イベント同期、Webhook受信Google Calendar API
integrations/csv_import.pyCSVインポート — パース、バリデーション、Bulk Insertファイルアップロード

各モジュールはCRMビジネスロジック層のサービス（communication.py、inquiry.py、phase.py、customer.py）を呼び出す構造とし、integrations → crm の一方向依存を維持する。逆方向の依存（crm → integrations）は禁止する。

エラーハンドリング共通パターン

すべての外部連携モジュールは以下の共通エラーハンドリングパターンに従う:

エラー種別ハンドリング方針具体例
認証エラー (401/403)トークンリフレッシュ試行 → 失敗時はテナント設定画面で再認証を促す通知LINE Access Token期限切れ、Gmail OAuth失効
レート制限 (429)Retry-Afterヘッダーに従ったバックオフLINE API送信上限、Gmail API割当超過
サーバーエラー (5xx)指数バックオフリトライ（最大3回）LINE Platform障害、Google API一時障害
パースエラーmanual_review ステータスで保存、担当者通知メールフォーマット変更、CSV不正データ
ネットワークエラー指数バックオフリトライ（最大3回）+ アラートタイムアウト、DNS解決失敗

リトライ戦略

外部API呼び出しには以下の指数バックオフリトライ戦略を適用する:

  • 最大リトライ回数: 3回
  • 初回待機時間: 1秒
  • バックオフ係数: 2（1秒 → 2秒 → 4秒）
  • ジッター: ±500ms のランダムジッターを追加（同時リトライの衝突回避）
  • リトライ対象: 5xxエラー、429エラー、ネットワークタイムアウト
  • リトライ非対象: 4xxエラー（401/429除く）、パースエラー、バリデーションエラー

3回のリトライがすべて失敗した場合は、エラーログを記録し、テナントのmanager以上にエスカレーション通知を送信する。Webhook受信（LINE/Calendar）の場合は200 OKを返却して再送を防止し、処理失敗を内部キューに記録して後続のバッチ処理でリカバリする。
