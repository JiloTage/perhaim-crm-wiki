CRM認証・権限詳細

最終更新: 2026-03-18
Per Heim CRM v3.2 における認証・権限設計の詳細仕様。既存のtenant_session_cookieによるログイン境界（company/store）を維持しつつ、3ロール（admin/manager/sales）の業務権限レイヤーを追加し、CrmActorオブジェクトによるきめ細やかなアクセス制御を実現する。

ロール拡張設計

既存の認証レイヤーは tenant_session_cookie の login_role（company または store）によるログイン境界のみを管理する。CRM統合にあたり、このログイン境界と業務権限を明確に分離するため、tenant_employees コレクションの各従業員レコードに role フィールドを追加し、admin・manager・sales の3ロールで業務権限を判定する。

ロール対象者データスコープ許可ログイン種別
admin本社管理者全店舗のデータ閲覧可（all_stores）company のみ
manager店長・マネージャー会社 or 自店舗（company_or_own_store）company / store
sales営業担当者自分の担当中心（assigned_only）company / store

設計上のポイント:

  • tenant_session_cookie は引き続き company | store のログイン境界として使用する（既存機能に変更なし）
  • CRMの業務権限は tenant_employees.role の値（admin | manager | sales）で判定する
  • 実際のデータアクセス範囲は login_role と employee.role の組み合わせで決定する
  • セッション埋め込み方式: ログイン時に employee_id + employee_role をセッショントークンに含め、サーバー側でセッションから取得する（X-Tenant-Employee-Id ヘッダーは廃止）

権限マトリクス

各ロールが各リソースに対して実行できる操作を以下のマトリクスで定義する。このPERMISSION_MATRIXはバックエンド側で crm_api.py のエンドポイントごとに参照され、CrmActor.employee_role に基づいてアクセス可否を判定する。

リソースadminmanagersales
顧客（customers）CRUD + merge + assignCRU + assignCRU
タスク（tasks）CRUD + assignCRU + assignCRU
コミュニケーション（communications）create + readcreate + readcreate + read
反響（inquiries）read + link + ignoreread + linkread
分析（analytics）readread-
設定（settings）read + updateread-
従業員管理（employees）CRUD--

補足事項:

  • merge: 顧客の重複統合。admin のみ実行可能（データ損失リスクがあるため）
  • assign: 担当者の割当・変更。admin と manager が実行可能
  • link: 反響レコードと既存顧客の紐づけ。admin と manager が実行可能
  • ignore: 反響レコードの無視（スパム等）。admin のみ実行可能
  • delete: 顧客・タスク・従業員の削除は admin のみ。誤操作防止のためソフトデリート推奨
  • コミュニケーション（LINE/メール/電話記録）は全ロールで作成・閲覧可能だが、データスコープ制御により閲覧範囲が異なる

データスコープ制御

権限マトリクスで操作種別を制御した上で、データスコープにより各ロールがアクセスできるデータの範囲を制限する。スコープは CrmActor の employee_role に基づいて自動決定される。

スコープ対象ロール動作仕様
all_stores admin テナント配下の全店舗のデータにアクセス可能。クエリに store_id フィルタを付与しない。分析画面では全店舗横断のKPIを表示する。
company_or_own_store manager login_role=company の場合は全店舗データにアクセス可能。login_role=store の場合は自店舗（CrmActor.store_id）のデータのみ。クエリに store_id フィルタを条件付きで付与する。
assigned_only sales 原則として自分が担当する顧客・タスクのみ。assigned_employee_id == CrmActor.employee_id でフィルタする。login_role=store の場合はさらに store_id でも制限される。未割当の反響は閲覧可能（自分で対応するため）。

CrmRepository — テナント分離多層防御

レビュー指摘 C4 + H4 対応: 従来は「全クエリに tenant_id フィルタ必須」という方針のみで強制メカニズムがなかった。CrmRepository の _base_query() により tenant_id を自動適用し、書き忘れを構造的に不可能にする。

全CRMクエリは CrmRepository の _base_query() 経由のみとし、生コレクション参照を禁止する。_scoped_query() はロールベースのデータスコープフィルタを追加で適用する。

# src/server/crm/repository.py — Phase 0 で実装
class CrmRepository:
    """テナント分離を構造的に強制するCRMリポジトリ基盤。
    全CRMクエリはこのクラス経由でのみ実行する。
    """
    def __init__(self, actor: CrmActor):
        self._actor = actor
        self._tenant_id = actor.tenant_id

    def _base_query(self, collection: str):
        """全クエリに tenant_id を自動適用（書き忘れ不可能）"""
        return get_db().collection(collection).where(
            "tenant_id", "==", self._tenant_id
        )

    def _scoped_query(self, collection: str):
        """data_scope を加味したクエリ（ロールベースフィルタ）"""
        q = self._base_query(collection)
        return self._actor.apply_scope(q)

    def list_customers(self, phase: str = None, ...):
        q = self._scoped_query("crm_customers")
        if phase:
            q = q.where("phase", "==", phase)
        return q.order_by("updated_at", direction="DESCENDING").stream()

    def get_customer(self, customer_id: str):
        doc = self._base_query("crm_customers").where(
            "__name__", "==", customer_id
        ).get()
        if not doc:
            raise NotFoundError(f"Customer {customer_id} not found")
        return doc[0].to_dict()

禁止事項: CRMコード内で get_db().collection() を直接呼び出すことは禁止。lint / code review で排除する。

スコープ判定ロジック（CrmActor.apply_scope() 内で適用）:

# CrmActor のスコープ適用メソッド
class CrmActor:
    # ...
    def apply_scope(self, query):
        """ロールに応じたデータスコープフィルタを適用"""
        scope = PERMISSION_MATRIX[self.employee_role]["data_scope"]

        if scope == "all_stores":
            # admin: tenant_id のみ（_base_query で適用済み）
            return query

        elif scope == "company_or_own_store":
            # manager: login_role に応じて分岐
            if self.login_role == "company":
                return query
            else:  # store login
                return query.where("store_id", "==", self.store_id)

        elif scope == "assigned_only":
            # sales: 自分の担当のみ
            base = query
            if self.login_role == "store":
                base = base.where("store_id", "==", self.store_id)
            return base.where("assigned_employee_id", "==", self.employee_id)

CrmActor解決フロー

レビュー指摘 C3 対応: X-Tenant-Employee-Id ヘッダーは廃止。セッショントークンに employee_id + employee_role を埋め込み、サーバー側でセッションから取得する方式に変更。

CRM APIは tenant_session_cookie に埋め込まれた employee_id と employee_role からCrmActorを構築する。ヘッダーによる操作者指定は廃止され、なりすましリスクを排除する。

resolve_tenant_actor() の処理手順

  1. セッション検証: request.cookies から tenant_session_cookie を取得し検証する。成功すると tenant_id、store_id、login_role、employee_id、employee_role が返る。
  2. Employee有効性確認: セッションに含まれる employee_id が現在も有効（active: true）であることをFirestoreで確認する。退職・無効化された従業員のアクセスを即座に遮断する。
  3. CrmActor構築: 検証済みの情報をもとに CrmActor を生成する。employee_role はセッションから取得し、未設定の場合はデフォルトで sales を適用する。
  4. ログイン種別チェック: PERMISSION_MATRIX[employee_role]["allowed_login_roles"] に login_role が含まれているかを検証する。例えば admin ロールは company ログインのみ許可されるため、store ログインからは 403 エラーとなる。
  5. 権限判定: 各エンドポイントで CrmActor.employee_role と PERMISSION_MATRIX を照合し、操作可否を判定する。
# CrmActor データ構造

認証フロー詳細

既存Cookie認証の継続利用

CRM機能の認証は既存の tenant_session_cookie をそのまま利用する。新たなログイン画面や認証フローは追加しない。ユーザーは既存の CompanyConsole または SalesConsole からログインし、CRM画面（/crm パス）にナビゲーションで遷移する。フロントエンドのルーティングでは RequireTenantAuth ガードの内側にCRMルートを配置し、未認証ユーザーのアクセスを防止する。

CRM APIでのセッション埋め込み認証

レビュー指摘 C3 対応: X-Tenant-Employee-Id ヘッダーは廃止。セッショントークンから employee_id を取得する。

CRM APIでは操作者の粒度が従業員レベルに細分化される。ログイン時にセッショントークンに employee_id + employee_role を埋め込み、サーバー側ではセッションからこれらの値を取得する。CRM では「1ログイン = 1従業員」モデルを採用する。

# crm_api.py - エンドポイント例（セッション埋め込み方式）
@router.get("/tenant/crm/customers")
def list_customers(request: Request):
    actor = resolve_tenant_actor(request)  # セッションから employee_id を取得
    crm_repo = CrmRepository(actor)        # テナント分離付きリポジトリ
    return crm_repo.list_customers(...)

フロントエンドの useCrmApi フックは、Cookie ベースの認証を自動的に使用する。employee_id はセッショントークンに含まれているため、フロントエンド側での明示的なヘッダー付与は不要。

セキュリティ考慮事項

脅威対策実装箇所
他人の employee_id を詐称（なりすまし） セッション埋め込み方式: employee_id はサーバー側でセッショントークンから取得。クライアントからの指定は不可能（C3 対応） resolve_tenant_actor()
テナント分離の書き忘れ CrmRepository._base_query() で全クエリに tenant_id を自動適用。生コレクション参照を禁止（C4 対応） CrmRepository
store ログインで admin 権限を取得 allowed_login_roles チェックで admin は company ログインのみに制限 crm_api.py エンドポイント
他店舗のデータへの不正アクセス データスコープ制御で store_id フィルタを自動適用 repository.py
Cookie 偽造・改ざん 既存の _require_tenant_session() による署名検証を継続利用 invite_sales.repository
権限昇格（role フィールド改ざん） role は Firestore サーバー側で管理。admin のみ employees リソースの CRUD が可能 PERMISSION_MATRIX
レビュー指摘 L3 対応: SameSite=Lax は GET リクエストの CSRF 保護が不十分（Top-level navigation で Cookie が送信される）。変更系 API（POST/PATCH/DELETE）で CSRF トークン検証を追加する。フロントエンドの useCrmApi hook で X-CSRF-Token ヘッダーを自動付与し、サーバー側で検証する。将来的に SameSite=Strict への移行を検討する。

Phase 0 実装における重要な設計判断:

  • invite_sales 側に公開ヘルパー resolve_tenant_actor() を追加し、private メソッド（_require_tenant_session、_require_operator_employee）の直接呼び出しを避ける
  • assigned_only スコープの判定は employee_id ベースで行い、Cookie 単独では許可しない
  • 将来的に RBAC からABAC（属性ベースアクセス制御）への拡張パスを確保するため、CrmActor に属性を追加しやすいデータクラス設計とする

並行運用時のRBACバイパス防止

レビュー指摘 H10 対応: SalesConsole/CRM 並行運用で RBAC バイパス可能。CRM 有効テナントでは sales ロールの /sales アクセスを制限し、feature flag で制御する。

CRM導入後も既存の SalesConsole（/sales）は一定期間並行運用される。この間、CRM側で定義した RBAC（admin/manager/sales）による権限制御が、SalesConsole 経由でバイパスされるリスクがある。具体的には、CRM側で sales ロールに制限している操作（例: 反響の ignore、顧客 merge）が、SalesConsole の既存APIから実行可能になるケースを防止する必要がある。

feature flag による制御

テナント設定に crm_enabled feature flag を導入し、CRM有効テナントでは以下の制御を適用する:

条件crm_enabled = truecrm_enabled = false
sales ロールの /sales アクセス 読み取り専用に制限。変更系操作はCRM画面（/crm）経由のみ許可する 従来通り全操作可能
manager / admin の /sales アクセス 従来通り（管理目的のアクセスを許可） 従来通り
CRM API（/v1/tenant/crm/*） RBAC に従いアクセス制御 404 Not Found を返却

バイパス防止の実装方針

  • ミドルウェアレベル: crm_enabled フラグが有効なテナントでは、/sales 配下の変更系エンドポイント（POST/PATCH/DELETE）に対し、sales ロールからのアクセスを 403 Forbidden で拒否する
  • フロントエンド: crm_enabled テナントの SalesConsole 画面では、変更系ボタン（編集・削除等）を非表示にし、CRM画面への誘導バナーを表示する
  • 段階的移行: テナントごとに crm_enabled を切り替えることで、段階的にCRMへ移行する。全テナント移行完了後、SalesConsole を廃止する
  • 監査ログ: crm_enabled テナントで /sales への変更系リクエストが拒否された場合、監査ログに記録してバイパス試行を追跡する
