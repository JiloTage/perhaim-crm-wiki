CRM認証・権限詳細

最終更新: 2026-03-18
Per Heim CRM v3.2 における認証・権限設計の詳細仕様。既存のtenant_session_cookieによるログイン境界（company/store）を維持しつつ、3ロール（admin/manager/sales）の業務権限レイヤーを追加し、CrmActorオブジェクトによるきめ細やかなアクセス制御を実現する。

ロール拡張設計

既存の認証レイヤーは tenant_session_cookie の login_role（company または store）によるログイン境界のみを管理する。CRM統合にあたり、このログイン境界と業務権限を明確に分離するため、tenant_employees コレクションの各従業員レコードに role フィールドを追加し、admin・manager・sales の3ロールで業務権限を判定する。

ロール対象者データスコープ許可ログイン種別
admin本社管理者全店舗のデータ閲覧可（all_stores）company のみ
manager店長・マネージャー会社 or 自店舗（company_or_own_store）company / store
sales営業担当者自分の担当中心（assigned_only）company / store

設計上のポイント:

  • tenant_session_cookie は引き続き company | store のログイン境界として使用する（既存機能に変更なし）
  • CRMの業務権限は tenant_employees.role の値（admin | manager | sales）で判定する
  • 実際のデータアクセス範囲は login_role と employee.role の組み合わせで決定する
  • CRM APIはすべて X-Tenant-Employee-Id ヘッダーを要求し、該当employeeがセッションに対して正当かを検証する

権限マトリクス

各ロールが各リソースに対して実行できる操作を以下のマトリクスで定義する。このPERMISSION_MATRIXはバックエンド側で crm_api.py のエンドポイントごとに参照され、CrmActor.employee_role に基づいてアクセス可否を判定する。

リソースadminmanagersales
顧客（customers）CRUD + merge + assignCRU + assignCRU
タスク（tasks）CRUD + assignCRU + assignCRU
コミュニケーション（communications）create + readcreate + readcreate + read
反響（inquiries）read + link + ignoreread + linkread
分析（analytics）readread-
設定（settings）read + updateread-
従業員管理（employees）CRUD--

補足事項:

  • merge: 顧客の重複統合。admin のみ実行可能（データ損失リスクがあるため）
  • assign: 担当者の割当・変更。admin と manager が実行可能
  • link: 反響レコードと既存顧客の紐づけ。admin と manager が実行可能
  • ignore: 反響レコードの無視（スパム等）。admin のみ実行可能
  • delete: 顧客・タスク・従業員の削除は admin のみ。誤操作防止のためソフトデリート推奨
  • コミュニケーション（LINE/メール/電話記録）は全ロールで作成・閲覧可能だが、データスコープ制御により閲覧範囲が異なる

データスコープ制御

権限マトリクスで操作種別を制御した上で、データスコープにより各ロールがアクセスできるデータの範囲を制限する。スコープは CrmActor の employee_role に基づいて自動決定される。

スコープ対象ロール動作仕様
all_stores admin テナント配下の全店舗のデータにアクセス可能。クエリに store_id フィルタを付与しない。分析画面では全店舗横断のKPIを表示する。
company_or_own_store manager login_role=company の場合は全店舗データにアクセス可能。login_role=store の場合は自店舗（CrmActor.store_id）のデータのみ。クエリに store_id フィルタを条件付きで付与する。
assigned_only sales 原則として自分が担当する顧客・タスクのみ。assigned_employee_id == CrmActor.employee_id でフィルタする。login_role=store の場合はさらに store_id でも制限される。未割当の反響は閲覧可能（自分で対応するため）。

スコープ判定ロジック（repository.py 内で適用）:

# データスコープのクエリフィルタ適用
def apply_scope_filter(query, actor: CrmActor):
    scope = PERMISSION_MATRIX[actor.employee_role]["data_scope"]

    if scope == "all_stores":
        # admin: tenant_id のみでフィルタ（全店舗アクセス可）
        return query.where("tenant_id", "==", actor.tenant_id)

    elif scope == "company_or_own_store":
        # manager: login_role に応じて分岐
        if actor.login_role == "company":
            return query.where("tenant_id", "==", actor.tenant_id)
        else:  # store login
            return query.where("tenant_id", "==", actor.tenant_id) \
                        .where("store_id", "==", actor.store_id)

    elif scope == "assigned_only":
        # sales: 自分の担当のみ
        base = query.where("tenant_id", "==", actor.tenant_id)
        if actor.login_role == "store":
            base = base.where("store_id", "==", actor.store_id)
        return base.where("assigned_employee_id", "==", actor.employee_id)

CrmActor解決フロー

CRM APIはCookie単独では操作者を確定しない。tenant_session_cookie（ログイン境界: company/store）と X-Tenant-Employee-Id（操作者識別）の2つを組み合わせて、CrmActor オブジェクトを構築する。

resolve_tenant_actor() の処理手順

  1. Cookie検証: request.cookies から tenant_session_cookie を取得し、_require_tenant_session(token) で検証する。成功すると tenant_id、store_id、login_role が返る。
  2. Employee検証: X-Tenant-Employee-Id ヘッダーの値を使い、_require_operator_employee(tenant_id, employee_id, store_id) で従業員レコードを取得・検証する。該当する従業員が存在しない、または店舗の所属が不一致であれば 403 エラーを返す。
  3. CrmActor構築: 検証済みの情報をもとに CrmActor を生成する。employee_role は従業員レコードの role フィールドから取得し、未設定の場合はデフォルトで sales を適用する。
  4. ログイン種別チェック: PERMISSION_MATRIX[employee_role]["allowed_login_roles"] に login_role が含まれているかを検証する。例えば admin ロールは company ログインのみ許可されるため、store ログインからは 403 エラーとなる。
  5. 権限判定: 各エンドポイントで CrmActor.employee_role と PERMISSION_MATRIX を照合し、操作可否を判定する。
# CrmActor データ構造
@dataclass
class CrmActor:
    tenant_id: str       # テナント識別子
    store_id: str        # 店舗識別子（store ログイン時に確定）
    login_role: str      # "company" | "store"（Cookie由来）
    employee_id: str     # 操作者の従業員ID（ヘッダー由来）
    employee_role: str   # "admin" | "manager" | "sales"（従業員レコード由来）

認証フロー詳細

既存Cookie認証の継続利用

CRM機能の認証は既存の tenant_session_cookie をそのまま利用する。新たなログイン画面や認証フローは追加しない。ユーザーは既存の CompanyConsole または SalesConsole からログインし、CRM画面（/crm パス）にナビゲーションで遷移する。フロントエンドのルーティングでは RequireTenantAuth ガードの内側にCRMルートを配置し、未認証ユーザーのアクセスを防止する。

CRM APIでのX-Tenant-Employee-Id必須化

既存の invite_sales APIは tenant_session_cookie のみで操作者を特定できるが、CRM APIでは操作者の粒度が従業員レベルに細分化されるため、X-Tenant-Employee-Id ヘッダーを全エンドポイントで必須とする。

# crm_api.py - エンドポイント例
@router.get("/tenant/crm/customers")
def list_customers(
    request: Request,
    operator_employee_id: str = Depends(_header_operator_employee_id),
):
    actor = resolve_tenant_actor(request, operator_employee_id)
    crm_repo = get_crm_repo()
    return crm_repo.list_customers(actor=actor, ...)

フロントエンドの useCrmApi フックは、ログイン後にローカルストレージまたはコンテキストから employee_id を取得し、すべてのCRM APIリクエストに X-Tenant-Employee-Id ヘッダーを自動付与する。

セキュリティ考慮事項

脅威対策実装箇所
他人の employee_id を詐称 _require_operator_employee() で Cookie セッションの tenant_id・store_id と従業員レコードの所属を照合 resolve_tenant_actor()
store ログインで admin 権限を取得 allowed_login_roles チェックで admin は company ログインのみに制限 crm_api.py エンドポイント
他店舗のデータへの不正アクセス データスコープ制御で store_id フィルタを自動適用 repository.py
Cookie 偽造・改ざん 既存の _require_tenant_session() による署名検証を継続利用 invite_sales.repository
権限昇格（role フィールド改ざん） role は Firestore サーバー側で管理。admin のみ employees リソースの CRUD が可能 PERMISSION_MATRIX

Phase 0 実装における重要な設計判断:

  • invite_sales 側に公開ヘルパー resolve_tenant_actor() を追加し、private メソッド（_require_tenant_session、_require_operator_employee）の直接呼び出しを避ける
  • assigned_only スコープの判定は employee_id ベースで行い、Cookie 単独では許可しない
  • 将来的に RBAC からABAC（属性ベースアクセス制御）への拡張パスを確保するため、CrmActor に属性を追加しやすいデータクラス設計とする
