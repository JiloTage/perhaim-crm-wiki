PII暗号化設計

最終更新: 2026-03-18
Per Heim CRM v3.2における個人識別情報（PII）の暗号化設計。AES-256-GCMによる暗号化、検索用ハッシュ生成、重複検知メカニズム、ロール別復号権限制御、および宅建業法に基づくデータ保持ポリシーを包括的に定義する。

本記事では、CRMデータモデル設計で定義された顧客モデル（Customer）および反響モデル（Inquiry）におけるPII（Personally Identifiable Information：個人識別情報）の暗号化設計を詳述する。Per Heim CRMは不動産業界特有の法規制（宅建業法）と個人情報保護法に準拠しつつ、高速な検索・重複検知を両立するため、暗号化＋検索用ハッシュの二重構造を採用している。本設計はPer Heim CRM v3.2 統合設計書のセキュリティ要件に基づき、CRM認証・権限詳細のロールモデルと連携して運用される。

PII暗号化の背景と要件

不動産CRMは顧客の電話番号、メールアドレス、反響メール原文など機微な個人情報を大量に扱う。宅建業法（宅地建物取引業法）は取引に関わる情報の適切な管理を義務づけており、個人情報保護法は個人データの安全管理措置を求めている。Per Heim CRMでは、これらの法規制に適合するため、PIIを保存時暗号化（encryption at rest）する方針を採る。

暗号化対象フィールド

暗号化の対象となるフィールドは以下の通りである:

モデルフィールド暗号化列検索用ハッシュ列説明
Customerphonephone_encryptedphone_search_hash電話番号
Customeremailemail_encryptedemail_search_hashメールアドレス
Inquiryphonephone_encryptedphone_search_hash反響から抽出した電話番号
Inquiryemailemail_encryptedemail_search_hash反響から抽出したメール
Inquiryraw_email_bodyraw_email_body_encrypted—反響メール原文

API入力では平文の phone / email を受け取ることが許可されるが、Firestoreへの永続化時点で必ず暗号化列と検索用ハッシュへ変換される。平文がデータベースに保存されることはない。

暗号化方式

AES-256-GCMの選定理由

Per Heim CRMでは暗号化アルゴリズムとしてAES-256-GCM（Galois/Counter Mode）を採用する。選定理由は以下の通りである:

  • 認証付き暗号化: GCMモードは暗号化と同時にメッセージ認証コード（MAC）を生成し、データの機密性と完全性を同時に保証する
  • 業界標準: NIST SP 800-38Dで標準化され、TLS 1.3の必須暗号スイートにも含まれる広く信頼されたアルゴリズム
  • 高速性: ハードウェアアクセラレーション（AES-NI）に対応し、大量レコードの一括暗号化/復号にも十分な性能を発揮する
  • Cloud KMS互換: Google Cloud KMSがネイティブにサポートしており、鍵管理との統合が容易

ノンス（Nonce）生成

AES-GCMの安全性を確保するため、暗号化ごとに12バイト（96ビット）のランダムノンスを生成する。同一鍵でノンスが再利用されると暗号の安全性が破壊されるため、以下の対策を講じる:

  • os.urandom(12) による暗号論的安全な乱数生成を使用
  • ノンスは暗号文の先頭12バイトに連結して保存（別カラム管理は不要）
  • 保存形式: base64(nonce[12] + ciphertext + auth_tag[16])

鍵管理（Cloud KMS連携）

暗号化鍵はアプリケーションコード内にハードコードせず、Google Cloud KMSで一元管理する:

  • エンベロープ暗号化: Cloud KMSのマスターキー（KEK）でデータ暗号化キー（DEK）をラップし、DEKで実際のPIIを暗号化する
  • 自動ローテーション: KEKは90日ごとに自動ローテーション。既存データは次回アクセス時に新キーで再暗号化
  • アクセス制御: KMSキーへのアクセスはIAMロールで厳格に制限。Cloud Runサービスアカウントのみが復号権限を持つ
  • 監査ログ: 鍵の使用はCloud Audit Logsに自動記録され、不正アクセスを検知可能

検索用ハッシュ設計

暗号化されたフィールドは直接検索できないため、検索用ハッシュを別フィールドに保存する。ハッシュは一方向性であり、ハッシュ値から元の平文を復元することはできない。

ハッシュ生成アルゴリズム

検索用ハッシュにはHMAC-SHA256を使用する。単純なSHA-256ではなくHMACを採用する理由は、レインボーテーブル攻撃への耐性を確保するためである。HMACの秘密鍵もCloud KMSで管理する。

search_hash = HMAC-SHA256(key=hmac_secret, message=normalized_value)
# 出力: 64文字のhex文字列

電話番号正規化ルール

同一の電話番号が異なる表記で登録された場合でもハッシュが一致するよう、ハッシュ生成前に以下の正規化処理を適用する:

ステップ処理入力例出力例
1全角→半角変換０９０−１２３４−５６７８090-1234-5678
2ハイフン・スペース・括弧除去090-1234-567809012345678
3国番号統一（+81→0）+81901234567809012345678
4先頭0の補完901234567809012345678

正規化後の電話番号は数字のみ11桁（携帯）または10桁（固定電話）となる。この正規化済み文字列に対してHMAC-SHA256を適用する。

メール正規化ルール

メールアドレスの正規化は以下の手順で行う:

ステップ処理入力例出力例
1全体を小文字化User@Example.COMuser@example.com
2前後の空白除去 user@example.com user@example.com
3全角→半角変換ｕｓｅｒ＠ｅｘａｍｐｌｅ．ｃｏｍuser@example.com

Gmailのドット無視ルール（u.ser@gmail.com = user@gmail.com）やサブアドレス（user+tag@gmail.com）については、Phase 1では正規化対象外とし、Phase 5の重複検知高度化で対応する。

重複検知メカニズム

反響受信時やCSVインポート時、同一顧客の二重登録を防止するためにsearch_hashベースの重複検知を行う。平文での一致検索は一切行わない。

重複候補検出フロー

  1. 新規顧客（または反響）の phone / email を受け取る
  2. 正規化処理を適用し、phone_search_hash / email_search_hash を生成
  3. Firestoreクエリで既存の crm_customers コレクションに対し、同一テナント内でハッシュ一致するドキュメントを検索: 
    # 電話番号での重複検索
db.collection("crm_customers")
  .where("tenant_id", "==", tenant_id)
  .where("phone_search_hash", "==", new_phone_hash)

# メールアドレスでの重複検索
db.collection("crm_customers")
  .where("tenant_id", "==", tenant_id)
  .where("email_search_hash", "==", new_email_hash)
  4. 一致が見つかった場合、duplicate_candidate_ids に候補顧客IDを記録
  5. 1件のみ一致かつ電話・メール両方が一致する場合は自動紐付け（auto_created）
  6. 複数候補または片方のみ一致の場合は手動レビュー（manual_review / duplicate）に回す

重複候補検索API

手動での重複確認には専用APIエンドポイントを使用する:

GET /v1/tenant/crm/customers/{customer_id}/duplicates
Response: { candidates: Customer[] }

このAPIは対象顧客の phone_search_hash / email_search_hash を用いて同一テナント内の候補を返却する。顧客マージは別エンドポイント（POST /v1/tenant/crm/customers/{customer_id}/merge）で実行する。

復号権限制御

PIIの復号は全ユーザーに無条件に許可されるわけではない。CRM認証・権限詳細で定義されたロールモデルに基づき、ロール別の復号可否を制御する。

ロール別復号権限

ロール復号可否対象範囲APIレスポンス例（phone）
admin全復号テナント内の全顧客"09012345678"
manager条件付き復号自店舗（store_id一致）の顧客のみ"09012345678" or "090-****-5678"
sales条件付き復号自身が担当（assigned_employee_id一致）の顧客のみ"09012345678" or "090-****-5678"
viewer不可—"090-****-5678"

マスキングルール

復号権限がないユーザーへのAPIレスポンスでは、以下のマスキング処理を適用する:

  • 電話番号: 先頭3桁と末尾4桁を表示、中間を **** に置換（例: 090-****-5678）
  • メールアドレス: ローカルパートの先頭2文字とドメインを表示（例: us****@example.com）
  • 反響メール原文: 復号権限がない場合はnullを返却

マスキングはAPIレスポンスのシリアライゼーション層で実施され、ビジネスロジック層には平文/暗号文のいずれも渡さない。

データ保持と削除

不動産取引に関わる個人情報の保持期間は、宅建業法および個人情報保護法の要件に準拠して設定する。

保持期間

データ区分保持期間根拠削除方式
顧客データ（成約: contracted）10年宅建業法（帳簿保存義務）保持期間満了後に物理削除
顧客データ（失注: lost）3年CRM v3.2 仕様保持期間満了後に物理削除
コミュニケーション記録5年宅建業法顧客削除に連動
監査ログ5年CRM v3.2 仕様自動物理削除
反響メール原文1年データ最小化原則パース済みデータは保持、原文のみ削除

論理削除と物理削除の使い分け

  • 論理削除（DELETE /v1/tenant/crm/customers/{customer_id}）: deleted_at タイムスタンプを設定。UIからは非表示となるが、データは保持期間満了まで残存する。管理者による復元が可能。
  • 物理削除: 保持期間満了後にバッチジョブで実行。暗号化列・検索用ハッシュ・関連するタイムラインイベント・コミュニケーション記録を含む全関連データを完全消去する。
  • 暗号化シュレッディング: 緊急時には暗号化キー（DEK）を破棄することで、暗号文を復号不能にする。データを個別に削除するより高速かつ確実。

GDPR対応

現時点でPer Heim CRMの主要市場は国内であるが、将来的なGDPR対応に向けて以下の設計方針を採用している:

  • データポータビリティ: 顧客データの復号・エクスポート機能を管理者向けに提供可能な構造
  • 忘れられる権利: 暗号化シュレッディングにより、削除要求への即座対応が可能
  • 同意管理: 将来のPhaseで同意取得・撤回フローを追加する拡張ポイントを確保
