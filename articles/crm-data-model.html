CRMデータモデル設計

最終更新: 2026-03-18
Per Heim CRM v3.2におけるFirestoreコレクション設計と各データモデル（顧客・タスク・タイムライン・コミュニケーション・反響）の詳細仕様。既存lead構造からcustomerモデルへのマイグレーション戦略を含む。

本記事では、Per Heim CRM v3.2 統合設計書で定義されたデータモデル設計の詳細を解説する。CRM機能の中核となるFirestoreコレクション設計、顧客（Customer）モデル、タスク（CrmTask）モデル、タイムライン（TimelineEvent）モデル、コミュニケーション（Communication）モデル、反響（Inquiry）モデルの各スキーマ定義と、既存のleadデータからcustomerへのマイグレーション戦略を包括的に記述する。

データモデル設計の基本方針は、既存の単一ドキュメント方式（invite_sales_state/default）からの脱却であり、CRM機能ではFirestoreのネイティブコレクションを採用する。これにより、大量データへのスケーラビリティ、クエリの柔軟性、セキュリティルールの細粒度制御が実現される。

Firestoreコレクション設計

既存のPer Heimシステムは invite_sales_state/default という単一ドキュメントにすべてのデータを格納する方式で動作している。しかし、CRMのデータ量増加（数千〜数万件の顧客レコード、膨大なタイムラインイベント）に対してこの方式は耐えられない。CRM機能ではFirestoreのネイティブコレクションを使い、各レコードを独立したドキュメントとして管理する。

前提条件

  • CRM機能を有効化する環境では INVITE_SALES_STORAGE_BACKEND=firestore を必須化する
  • local/dev は Firestore Emulator を使用。JSON backend は既存 invite_sales のみを対象とするレガシー運用とし、CRMの正式サポート対象外とする
  • 既存の invite_sales_state/default と CRM コレクション群が同一 Firestore 上で一貫して参照可能

コレクション構造

新規に追加するコレクション群は以下の通りである:

コレクションパス区分説明

crm_customers/{customer_id}新規顧客マスタ
crm_tasks/{task_id}新規タスク管理
crm_timeline/{event_id}新規タイムラインイベント

crm_settings サブコレクション設計

レビュー指摘 M4 対応: crm_settings の 1 ドキュメント肥大化リスクに対し、サブコレクション化を推奨する。

crm_settings/{tenant_id} ドキュメントに LINE 連携設定・Gmail 連携設定・カレンダー設定・通知設定などすべてのテナント設定を格納すると、ドキュメントサイズが Firestore の 1MB 上限に近づくリスクがある。また、設定カテゴリごとのアクセス制御やキャッシュ管理が困難になる。以下のサブコレクション構造を推奨する:

パス用途主なフィールド
crm_settings/{tenant_id}（ルート）基本設定のみcrm_enabled, plan, timezone, wrapped_dek
crm_settings/{tenant_id}/line_configLINE 連携設定channel_id, channel_secret, webhook_url, rich_menu_id
crm_settings/{tenant_id}/gmail_configGmail 連携設定service_account, watch_label, parser_rules, forwarding_address
crm_settings/{tenant_id}/calendar_configカレンダー連携設定google_calendar_id, sync_enabled, default_event_duration
crm_settings/{tenant_id}/notification_config通知設定inquiry_notify_channels, task_reminder_timing, escalation_rules

ルートドキュメント（crm_settings/{tenant_id}）には CRM の有効/無効フラグやプラン情報など基本的な設定のみを保持し、各チャネル連携やカテゴリ別の設定はサブコレクション内の個別ドキュメントに分離する。これにより:

  • ドキュメントサイズの肥大化を防止し、Firestore の 1MB 制限に余裕を持たせる
  • 設定カテゴリごとの Firestore Security Rules を細粒度で設定可能
  • 各設定の独立した更新・キャッシュ管理が可能（LINE 設定変更時に全設定の再読み込みが不要）
  • 将来的な設定カテゴリ追加時にルートドキュメントへの影響がない

ニーズ構造タグ（NeedsTags）

NeedsTagsはPer Heim CRMの差別化コア機能である。AIヒアリングから抽出された顧客ニーズを4階層の優先度で構造化し、物件マッチングや追客戦略の最適化に活用する。

フィールド型説明例
dealbreakerslist[string]絶対NG条件["1階不可", "線路沿い不可"]
must_haveslist[string]必須条件["2LDK以上", "駅徒歩10分以内"]
nice_to_haveslist[string]あったら嬉しい条件["宅配ボックス", "オートロック"]
dont_carelist[string]気にしない条件["築年数", "向き"]

PII暗号化設計

レビュー指摘 H6 対応: 検索用ハッシュは SHA-256 単独ではなく HMAC-SHA256 + 専用 secret を使用する。rainbow table による逆引きを防止するため。

個人情報保護のため、顧客の連絡先情報には暗号化＋検索用ハッシュの二重構造を採用する。詳細な暗号化方式とキー管理についてはPII暗号化設計を参照。

  • 保存ルール: API入力は平文の phone / email を受け取ってよい
  • 暗号化処理: リポジトリ保存時に phone_encrypted / email_encrypted と phone_search_hash / email_search_hash を同時生成
  • 検索用ハッシュ: HMAC-SHA256(正規化済みPII, search_hash_secret) で生成。secret は PII 暗号化鍵とは分離管理する
  • 検索: 一覧検索・重複検知はハッシュ列を使用
  • 表示: 画面表示時のみ復号する

自動生成ルール

auto_generated: true のタスクは、以下のようなイベントをトリガーとしてシステムが自動作成する:

  • 反響受信時: 「初回連絡」タスクを自動作成（priority: high）
  • ヒアリング完了時: 「ヒアリング結果確認・来店提案」タスクを自動作成
  • フェーズ遷移時: 遷移先フェーズに応じた次アクションタスクを自動作成
  • 期限超過時: リマインダータスクを再生成

タイムラインモデル（TimelineEvent）

TimelineEventは顧客に紐づく全接点の時系列記録を担う。営業活動の透明性と追跡可能性を確保するために、あらゆる顧客接点がイベントとして記録される。

タイムラインのスケーラビリティ対策

レビュー指摘 M2 対応: crm_timeline コレクションの肥大化に対する対策を定義する。長期運用でイベント数が増加しても、パフォーマンスとコストを維持するための戦略。

タイムラインイベントは顧客接点のたびに蓄積されるため、長期運用では1顧客あたり数百〜数千件に達する可能性がある。以下の対策を組み合わせてスケーラビリティを確保する。

ページネーション（Cursor-based）

タイムライン取得APIでは cursor-based pagination を採用する。created_at DESC の複合インデックスを使用し、最新N件を効率的に取得する。

  • デフォルト取得件数: 最新20件
  • カーソル: startAfter(last_document) で次ページを取得
  • インデックス: tenant_id + customer_id + created_at (DESC)
  • APIパラメータ: ?cursor=xxx&limit=20

アーカイブ戦略

6ヶ月以上前のタイムラインイベントは crm_timeline_archive コレクションに移動し、アクティブコレクションのサイズを抑制する。

  • アーカイブ対象: created_at が6ヶ月以上前のイベント
  • アーカイブ先: crm_timeline_archive/{event_id} コレクション（同一スキーマ）
  • 実行方式: Cloud Scheduler による日次バッチ（毎日 03:00 JST）
  • 処理フロー: 対象イベントを batch write でアーカイブ先にコピー → 元コレクションから削除（トランザクション）
  • UIでのアーカイブ参照: 顧客カード画面に「過去の履歴を表示」リンクを設け、アーカイブコレクションから追加読み込み可能にする

lead → customer マイグレーション

既存の lead データを新しい Customer モデルに変換するマイグレーションは、CRM導入の最も重要な初期作業である。

前提と制約

  • 現行leadには phone / email フィールドが存在しないため、初回マイグレーション時点では厳密な「同一人物統合」は不可能
  • 初回移行は暫定的に 1 lead = 1 customer とする
  • 複数leadを1 customerにまとめる統合は、Phase 1 の merge UI と Phase 5 の重複検知導入後に段階的に実施

フェーズマッピング

既存leadの status を新しい CustomerPhase に以下のルールでマッピングする:

lead.statuscustomer.phase説明
newhearing_pending新規作成 → ヒアリング待ち
openedhearing_activeURL開封済み → ヒアリング中
in_progresshearing_active進行中 → ヒアリング中
completedhearing_done完了 → ヒアリング済み
revokedlost取消 → 失注

ニーズタグ抽出

マイグレーション時、既存の artifact_record.content（AIヒアリング結果）からニーズ構造を抽出し、NeedsTags として構造化する。プロファイルの facts セクションから条件を分類し、dealbreakers / must_haves / nice_to_haves / dont_care の4カテゴリに振り分ける。

ID変換戦略と冪等性保証

レビュー指摘 H3 対応: マイグレーションの冪等性を保証するため、ULID ベースの customer_id と crm_migration_map による二重実行防止を導入する。

顧客IDは ULID を使用する: customer_id = "cust_" + ULID()。lead_id からの導出ではなく、新規にユニークIDを生成する。これにより将来の統合やマージ時のID衝突を防止する。

冪等性保証: crm_migration_map

マイグレーションの二重実行を防止するため、crm_migration_map コレクションでマッピングを管理する:

# crm_migration_map/{lead_id} ドキュメント
{
    "lead_id": "lead_xxx",
    "customer_id": "cust_01HXXXXXX",  # ULID
    "migrated_at": "2026-03-20T00:00:00+09:00",
    "migration_version": 1,
    "status": "completed"  # "pending" | "completed" | "failed"
}

# マイグレーション処理（冪等）
def migrate_lead_to_customer(lead_id: str) -> str:
    # 1. 既にマイグレーション済みか確認
    existing = db.collection("crm_migration_map").document(lead_id).get()
    if existing.exists and existing.to_dict()["status"] == "completed":
        return existing.to_dict()["customer_id"]  # 冪等: 既存IDを返す

    # 2. 新規 customer_id を ULID で生成
    customer_id = f"cust_{ulid.new()}"

    # 3. Firestore Transaction で原子的に実行
    @firestore.transactional
    def _migrate(transaction):
        # migration_map を作成
        transaction.set(
            db.collection("crm_migration_map").document(lead_id),
            {"lead_id": lead_id, "customer_id": customer_id, ...}
        )
        # customer を作成
        transaction.set(
            db.collection("crm_customers").document(customer_id),
            build_customer_from_lead(lead_id, customer_id)
        )

    _migrate(db.transaction())
    return customer_id

lead_ids 配列に元のlead_idを格納し、1:N関係の基盤を構築する。

GDPR/個人情報保護法対応: カスケード削除設計

レビュー指摘 M5 対応: 顧客データ削除要求時の関連データのカスケード削除フローを定義する。

GDPR（EU一般データ保護規則）の「忘れられる権利（Right to Erasure）」および日本の個人情報保護法における本人からの利用停止・消去請求に対応するため、顧客データ削除時のカスケード削除設計を定義する。Customer を起点として、関連する全データを漏れなく削除する仕組みが必要である。

カスケード削除対象

顧客削除要求を受けた場合、以下の関連データすべてが削除対象となる:

コレクション検索条件削除対象データ削除優先度
crm_customerscustomer_id 一致顧客マスタ（PII 暗号化フィールド含む）最後（参照整合性維持のため）
crm_timelinecustomer_id 一致全タイムラインイベント高
crm_taskscustomer_id 一致全タスク（完了・未完了問わず）高
crm_communicationscustomer_id 一致全コミュニケーション記録（LINE・メール・電話）高
crm_inquiriescustomer_id 一致紐付け済み反響レコード中（反響自体は匿名化して保持可能）

ソフトデリート設計

物理削除ではなくソフトデリート（deleted_at フィールド）を推奨する理由:

  • 誤削除の復旧: 一定期間内であれば管理者が削除を取り消し可能
  • 法的保持義務との両立: 宅建業法で求められる帳簿保存義務（成約データ10年）と削除要求を両立。法的義務がある場合は削除要求を拒否できるが、その判断期間を確保する
  • 監査証跡の維持: いつ・誰が・何を削除したかの記録を保持

各コレクションのドキュメントに以下のフィールドを追加する:

フィールド型説明
deleted_atstring | null論理削除日時（ISO 8601）。null = 未削除
deleted_bystring | null削除を実行した従業員ID
deletion_reasonstring | null削除理由（"gdpr_request" | "customer_request" | "data_retention_expired"）

物理削除バッチ

ソフトデリートされたデータは、保持期間満了後にバッチジョブで物理削除を実行する:

  • 実行タイミング: 日次バッチ（深夜帯）で deleted_at が保持期間を超過したドキュメントを検索
  • 保持期間: 削除要求から 30日間はソフトデリート状態を維持（誤削除対応猶予期間）
  • 物理削除処理: Firestore の batch.delete() で関連ドキュメントを一括削除。500件ずつコミットして Firestore のバッチ上限に対応
  • 監査ログ: 物理削除の実行記録を crm_audit_logs に記録。監査ログ自体は別途5年間保持
  • 暗号化シュレッディング: 緊急の完全削除が必要な場合は、対象テナントの DEK を破棄することで暗号文を復号不能にする（PII暗号化設計参照）
