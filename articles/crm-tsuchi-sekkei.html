通知設計詳細

最終更新: 2026-03-18
Per Heim CRM v3.2における通知基盤の詳細設計。3段階の通知チャネル（即時・日次・エスカレーション）、7種の通知イベント定義、NotificationServiceを中核とした処理アーキテクチャ、Web Pushによるリアルタイム通知、Cloud Tasks/Schedulerによるバッチ通知、およびユーザー単位の通知設定・制御機能を包括的にカバーする。

通知チャネル設計

CRM通知は、情報の緊急度と重要度に応じて3段階のチャネルに分類される。不動産営業では反響への即時対応が成約率に直結するため、チャネル設計は「営業が最速で行動できる」ことを最優先とする。

即時通知（Web Push + バッジ）

目的: 営業担当者に即座にアクションを促す。反響受信・ヒアリング完了・LINE受信など、顧客との接点発生を即時に伝達する。

項目仕様
配信方式Web Push API（Service Worker経由） + Firestore onSnapshot によるCRM画面内バッジ
遅延許容5秒以内
対象ブラウザChrome, Edge, Firefox（Safari は将来対応）
オフライン時Push通知はブラウザ再起動時に配信。バッジはFirestoreリアルタイム同期により次回アクセス時に反映
既読管理バッジクリック時に notifications/{id}.read_at を更新

日次通知（メール）

目的: 営業担当者の日次業務をサポートする。朝のタスクサマリーと夕方の未完了リマインダーにより、業務の開始と振り返りを自動化する。

項目仕様
配信方式Cloud Scheduler → Cloud Tasks → SendGrid API / Gmail API
テンプレートHTML + プレーンテキスト の multipart/alternative
差出人noreply@{tenant_domain} またはテナント設定のメールアドレス
配信停止メール末尾に1クリック配信停止リンク（List-Unsubscribeヘッダー付与）

エスカレーション通知（メール + Push）

目的: 対応遅延をマネージャーに自動エスカレーションする。反響未対応30分超過やタスク期限1日超過を検知し、管理者に即座に通知する。

項目仕様
配信方式Cloud Scheduler による定期チェック → 条件合致時に メール + Web Push
対象者該当テナントの manager ロールを持つ従業員
エスカレーション階層1次: 直属マネージャー → 2次（未対応1時間超過）: テナント管理者
重複抑止同一イベントに対し1時間以内の再通知を抑止（notification_cooldown）

チャネル比較サマリー

レベルチャネル実装方式遅延対象
即時 Web Push + バッジ Service Worker + Firestore onSnapshot <5秒 担当者
日次 メール Cloud Scheduler + SendGrid / Gmail API スケジュール 全従業員
エスカレーション メール + Push Cloud Scheduler + 条件判定ロジック チェック間隔依存 マネージャー

通知イベント定義

CRM通知イベントは7種定義され、即時通知（3種）、日次バッチ（2種）、エスカレーション（2種）に分類される。各イベントはPython辞書 NOTIFICATION_EVENTS として一元管理し、CRM API設計のエンドポイントから発火される。

即時通知イベント

inquiry_received（新規反響受信）

項目仕様
トリガーSUUMOやHOME'Sからの反響メール取込完了時、または手動顧客登録時
対象者assigned_employee（割当済みの場合）または round_robin で自動選択
チャネルpush + badge
メッセージ新規反響: {customer_name} ({source})
ペイロード{ event: "inquiry_received", customer_id, source, assigned_to }
アクション通知クリックで顧客詳細画面に遷移

hearing_completed（AIヒアリング完了）

項目仕様
トリガーcomplete_session のCRMイベントフック発火時
対象者assigned_employee
チャネルpush + badge
メッセージ{customer_name}のAIヒアリングが完了しました
ペイロード{ event: "hearing_completed", customer_id, session_id, needs_tags[] }
アクション通知クリックで顧客カードのAIプロファイルタブに遷移

line_received（LINEメッセージ受信）

項目仕様
トリガーLINE Webhook受信 → user_id → customer_id マッピング解決後
対象者assigned_employee
チャネルpush + badge
メッセージ{customer_name}からLINEメッセージ
ペイロード{ event: "line_received", customer_id, line_user_id, preview }
アクション通知クリックでコミュニケーション画面のLINEタブに遷移

日次バッチイベント

daily_task_summary（日次タスクサマリー）

項目仕様
トリガーCloud Scheduler: 毎日 09:00 JST
対象者all_employees（テナント内全従業員）
チャネルemail
メッセージ本日のタスク: {pending_count}件
メール内容未完了タスクリスト（期限順）、本日の予定来店、フォローアップ対象顧客一覧
送信条件タスクが0件でも送信（「本日のタスクはありません」表示）

overdue_reminder（未完了タスクリマインダー）

項目仕様
トリガーCloud Scheduler: 毎日 17:00 JST
対象者assigned_employee（未完了タスクを持つ担当者のみ）
チャネルemail + badge
メッセージ未完了タスク: {overdue_count}件
メール内容期限超過タスクリスト（超過日数順）、各タスクの顧客名と期限日
送信条件未完了タスクがある場合のみ送信

エスカレーションイベント

inquiry_unresponded_30min（反響未対応30分超過）

項目仕様
トリガーCloud Scheduler: 5分間隔で未対応反響をチェック → created_at から30分超過を検知
対象者manager（該当店舗のマネージャー）
チャネルemail + push
メッセージ反響未対応30分超過: {customer_name}
判定条件顧客フェーズが new かつ first_response_at が NULL かつ created_at + 30分 < 現在時刻
クールダウン同一顧客に対し1時間以内の再通知を抑止

task_overdue_1day（タスク期限1日超過）

項目仕様
トリガーCloud Scheduler: 毎日 10:00 JST にチェック
対象者manager（該当タスク担当者の直属マネージャー）
チャネルemail
メッセージタスク期限1日超過: {task_title} (担当: {employee_name})
判定条件タスクステータスが pending / in_progress かつ due_date + 1日 < 現在日
クールダウン同一タスクに対し24時間以内の再通知を抑止

イベント名分類チャネル対象タイミング

hearing_completed即時push + badge担当者ヒアリング完了時

NotificationServiceアーキテクチャ

通知の送信はNotificationServiceクラスが一元管理する。すべての通知イベントはこのサービスを経由し、チャネルに応じた処理分岐（Web Push API / Firestore write / Cloud Tasks enqueue）を行う。この設計により、通知ロジックの集約と将来のチャネル追加（Slack連携など）を容易にする。

クラス設計

NotificationServiceは以下の責務を持つ:

  • イベント受信: CRM各モジュールからの通知リクエストを notify(event_type, context) メソッドで受け付ける
  • 対象者解決: イベント定義の target フィールドに基づき通知対象の従業員を解決する（assigned_employee, round_robin, manager, all_employees）
  • チャネル分岐: イベント定義の channel フィールドに基づき配信チャネルを決定する
  • 通知設定フィルタ: ユーザーの通知設定（ON/OFF、営業時間フィルタ）を適用する
  • クールダウン判定: 重複通知の抑止ロジックを適用する
  • 配信実行: チャネルごとのアダプターを呼び出して実際の通知を送信する
  • 配信記録: 全通知を crm_notifications/{tenant_id}/logs に記録する

エラーハンドリング

障害パターン対策リトライ
Web Push API 送信失敗 ステータスコード 410（Gone）の場合はサブスクリプション削除。それ以外はリトライ 最大3回、指数バックオフ（1s, 2s, 4s）
Firestore write 失敗 トランザクション再試行。永続的失敗はエラーログに記録 Firestore SDK の自動リトライに委譲
Cloud Tasks enqueue 失敗 即座にリトライ。失敗持続時はデッドレターキューに退避 最大5回、指数バックオフ
SendGrid API エラー 429（Rate Limit）はバックオフ後リトライ。5xx はリトライ。4xx はエラーログ Cloud Tasks のリトライポリシーに委譲（最大10回）

Web Push実装

Web Push APIを用いたブラウザ通知は、CRMのリアルタイム通知の中核をなす。Service Workerの登録からプッシュサブスクリプション管理、ペイロード設計まで、実装に必要な全仕様を定義する。

Service Worker登録

CRMフロントエンドの初期化時に Service Worker を登録する。

  • ファイル配置: frontend/public/sw-crm-push.js
  • 登録タイミング: CRM画面の初回アクセス時（CrmLayout コンポーネントの useEffect）
  • スコープ: /crm/ 配下のみ
  • 更新戦略: registration.update() をアプリ起動時に毎回呼び出し

Pushサブスクリプション管理

ユーザーごとの Push サブスクリプション情報を Firestore に管理する。

  • 保存先: crm_push_subscriptions/{tenant_id}/employees/{employee_id}/devices/{device_id}
  • 保存データ: endpoint, keys.p256dh, keys.auth, user_agent, created_at, last_used_at
  • 複数デバイス: 1従業員あたり最大5デバイスまで登録可能。超過時は last_used_at が最も古いものを自動削除
  • VAPID鍵管理: VAPID_PUBLIC_KEY / VAPID_PRIVATE_KEY を Secret Manager に格納
レビュー指摘 L4 対応: VAPID（Voluntary Application Server Identification）鍵ペアの管理方針を明確化する。公開鍵はフロントエンドの Service Worker に埋め込み、秘密鍵は GCP Secret Manager で管理する。鍵ローテーションは年1回を推奨する（ユーザーの再購読が必要なため慎重に実施すること）。

Cloud Tasks/Schedulerによるバッチ通知

Cloud Schedulerが定期的にHTTPトリガーを発火し、Cloud Tasksが個々のメール送信をキューイングする。この2層構造により、スケジュール管理とメール配信の責務を分離し、大量配信時のスロットリングとリトライを確実に行う。

Cloud Schedulerジョブ定義

ジョブ名スケジュール（cron）トリガーURL処理内容
daily_summary 0 9 * * * (JST) POST /api/crm/notifications/daily-summary 全テナントの全従業員に日次タスクサマリーメール送信
overdue_check 0 17 * * * (JST) POST /api/crm/notifications/overdue-check 未完了タスクを持つ担当者にリマインダーメール送信
monthly_report 0 9 1 * * (JST) POST /api/crm/notifications/monthly-report テナント管理者に月次サマリーレポートメール送信
escalation_check */5 9-18 * * 1-5 (JST) POST /api/crm/notifications/escalation-check 反響未対応30分超過チェック（平日9:00-18:00の5分間隔）
task_overdue_escalation 0 10 * * * (JST) POST /api/crm/notifications/task-overdue-escalation タスク期限1日超過をマネージャーにエスカレーション

バッチ通知処理フロー

Cloud Schedulerのトリガーを受けたCRM APIエンドポイントは以下の手順で処理する:

  1. 全テナントのリストを取得
  2. テナントごとに対象従業員をクエリ
  3. 従業員ごとにメール内容を生成
  4. Cloud Tasks キューにタスクを enqueue（1メール = 1タスク）
  5. Cloud Tasks Worker が SendGrid API を呼び出してメール送信
  6. 送信結果を crm_notifications/{tenant_id}/logs に記録

通知設定・ユーザー制御

通知の受信はユーザーが制御できる必要がある。テナントレベルと従業員レベルの2階層で通知設定を管理し、テナント設定が上位の制約として機能する。

テナントレベル設定

テナント管理者が設定する全社的な通知ポリシー。crm_settings/{tenant_id}/notification に保存する。

設定項目型デフォルト説明
push_enabledbooleantrueWeb Push通知の全社ON/OFF
email_enabledbooleantrueメール通知の全社ON/OFF
escalation_enabledbooleantrueエスカレーション通知のON/OFF
business_hours_startstring"09:00"営業時間開始（JST）
business_hours_endstring"18:00"営業時間終了（JST）
business_daysnumber[][1,2,3,4,5]営業曜日（0=日, 1=月, ...6=土）
escalation_threshold_minnumber30エスカレーションまでの未対応時間（分）

従業員レベル設定

各従業員が個人の通知受信を制御する。crm_settings/{tenant_id}/employees/{employee_id}/notification に保存する。テナントレベルで無効化されているチャネルは従業員レベルで有効化できない。

設定項目型デフォルト説明
push_enabledbooleantrueWeb Push通知の個人ON/OFF
email_enabledbooleantrueメール通知の個人ON/OFF
daily_summary_enabledbooleantrue日次タスクサマリーメールのON/OFF
overdue_reminder_enabledbooleantrue未完了リマインダーのON/OFF
quiet_hours_enabledbooleanfalse営業時間外の通知抑止
quiet_hours_startstring"20:00"個人静寂時間開始
quiet_hours_endstring"08:00"個人静寂時間終了

営業時間フィルタロジック

通知配信時の営業時間フィルタは以下の優先順位で適用する:

  1. 即時通知（エスカレーション含む）: 営業時間フィルタをバイパス（常に配信）
  2. 従業員の quiet_hours: 有効な場合、Push通知を抑止（メールは配信）
  3. テナントの business_hours: 営業時間外の日次バッチ通知は次の営業時間開始時にキューイング

通知設定画面

CRM設定画面（/crm/settings/notifications）にて以下を提供する:

  • 管理者向け: テナントレベルの全通知設定、エスカレーション閾値調整、営業時間設定
  • 従業員向け: 個人の通知ON/OFF切り替え、静寂時間設定、テスト通知送信ボタン
  • 通知履歴: 直近30日間の通知配信ログ一覧（ステータス、配信日時、チャネル）
