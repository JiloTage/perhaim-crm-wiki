## 0. エグゼクティブサマリー

設計の方向性 -- 「既存 invite_sales を壊さず CRM を独立モジュールとして追加」 -- は正しい。ただし **5件の Critical 指摘** と **11件の High 指摘** があり、そのまま Phase 1 に進むことはできない。

最大のリスクは「設計書が "フル CRM" を前提にしているが、GTM としてはまず "AI Hearing Dashboard" として出すべき」というプロダクト戦略の軸ずれ。技術的には、既存の単一ドキュメントアーキテクチャとの接続設計、認証のなりすましリスク、テナント分離の多層防御不足が Critical。

**Go 条件**: Phase 0 の設計修正（5-8日）で Critical の大部分を解消可能。PII 暗号化のみ Phase 4 まで延期可。

## 1. 設計の強み

| # | 評価点 |
|---|--------|
| 1 | **既存コード非破壊方針**: invite_sales を「読み取り専用参照 + イベントフック」で CRM と接続する設計は、既存テナントへの影響を最小化する正しい判断 |

## 2. Critical 指摘事項 (Go ブロッカー: 5件)

### C1. invite_sales 単一ドキュメントとの接続設計が不十分
**発見**: backend-reviewer
**問題**: CRM から invite_sales の `_db` を参照する際、threading lock 競合・不整合データ読み取り・1MB ドキュメント全取得のリスクがある。設計書の「読み取り専用参照」は具体策なし。

**合意された解決策**:
- invite_sales に公開メソッド `get_lead_snapshot()`, `get_artifacts_snapshot()` を追加（lock 内で deepcopy）
- CRM バックエンドはこのメソッド経由でのみ invite_sales データにアクセス
- フロントエンドは `crm_customers` コレクションの onSnapshot を使い、`invite_sales_state/default` は直接 watch しない
- Phase 0 で `invite_sales/public_api.py` として公開インターフェースを整備

### C2. complete_session() イベントフック設計不足
**発見**: backend-reviewer
**問題**: ヒアリング完了時の CRM 通知メカニズムが未設計。lock 内 CRM 書き込みで deadlock、失敗時リトライ未定義。

**合意された解決策**:
- **Cloud Tasks** による非同期フック（lock の外側で enqueue）
- `/v1/internal/crm/on-hearing-complete` 内部エンドポイント（内部トークン認証）
- リトライ: exponential backoff 5回、dead letter queue で管理者通知
- **Idempotency key**: `crm_hearing:{tenant_id}:{lead_id}:{session_id}`（replay 攻撃防止）
- フェーズ遷移は**条件付き**（現在の phase が `hearing_active` の場合のみ `hearing_done` に遷移、楽観的ロック）
- `HearingCompletionEvent` Pydantic モデルを Phase 0 で定義（F3-2 API スキーマ v1 = Go Condition #2 の解消）

### C3. X-Tenant-Employee-Id ヘッダーがなりすまし可能
**発見**: security-reviewer, backend-reviewer
**問題**: ヘッダーは平文文字列。同一テナントの別従業員のIDを自由にセットでき、監査ログ・権限チェックが無意味になる。

**合意された解決策**:
- **セッション埋め込み方式**: ログイン時に `employee_id` + `employee_role` をセッショントークンに含める
- `X-Tenant-Employee-Id` ヘッダーは廃止し、サーバー側でセッションから employee_id を取得
- CRM では「1ログイン = 1従業員」モデル（CRM v3.2 の F8-4 per-employee 認証に合致）
- Phase 0 の `resolve_tenant_actor()` 実装に含める

### C4. テナント分離に多層防御がない
**発見**: security-reviewer, backend-reviewer
**問題**: 「全クエリに tenant_id フィルタ必須」は方針のみで、強制メカニズムがない。1つのクエリの書き忘れで全テナントのデータが漏洩。

**合意された解決策**:
```python
class CrmRepository:
    def __init__(self, actor: CrmActor):
        self._actor = actor
        self._tenant_id = actor.tenant_id

    def _base_query(self, collection: str):
        """全クエリに tenant_id を自動適用（書き忘れ不可能）"""
        return get_db().collection(collection).where("tenant_id", "==", self._tenant_id)

    def _scoped_query(self, collection: str):
        """data_scope を加味したクエリ（ロールベースフィルタ）"""
        q = self._base_query(collection)
        return self._actor.apply_scope(q)
```
- 全 CRM クエリは `_base_query()` 経由のみ（生コレクション参照を禁止）
- CRM コード内で `get_db().collection()` 直接呼び出しを lint/code review で排除
- Phase 0 で CrmRepository 基盤として実装

### C5. lead↔customer リアルタイム同期メカニズムが未定義
**発見**: backend-reviewer + frontend-reviewer（クロスレビュー）
**問題**: SalesConsole と CRM の並行運用期間中、lead.status と customer.phase の同期が未設計。手動フェーズ変更、ヒアリング完了、revoke 等のイベントが片方にしか反映されない。

**合意された解決策: 段階移行方式**
- Phase 1: CRM は invite_sales を読み取り専用参照。customer が source of truth。
- Phase 3: SalesConsole のステータス変更を CRM の phase transition API 経由に切替。CRM API が内部的に `crm_customers.phase` と `invite_sales.leads.status` の両方を更新。
- Phase 7+: SalesConsole の機能を CRM に統合完了。
- SalesConsole に「この顧客は CRM で管理されています [CRM で見る →]」バナーを表示。
- 設計書 Section 4.4 に「並行運用期間の同期戦略」を追記すべき。

## 3. High 指摘事項 (Phase 0-1 で解決必須: 11件)

| # | 指摘 | 発見者 | 解決策 |
|---|------|--------|--------|
| H1 | Firestore コンポジットインデックス不足 | backend | crm_communications, crm_inquiries, 重複検知用の phone/email_search_hash インデックス追加。Phase 1 では最小 4-5本 |
| H2 | PerHeim Sync v2 パターンとの不整合 | backend | CRM → PerHub 方向の sync 設計、idempotency_key 統合方針を Section 9 に追記 |
| H3 | マイグレーション「1 lead = 1 customer」の冪等性 | backend | customer_id は ULID 使用、crm_migration_map で冪等性保証、二重実行防止 |
| H4 | 権限マトリクスが Firestore クエリレベルで強制されない | backend + security | `_scoped_query()` で role ベースフィルタを自動適用（C4 の解決策に統合済み） |
| H5 | フェーズ遷移のトランザクション設計なし | backend + security | customer update + timeline create + task create を 1 Firestore transaction で実行。race condition 防止 |
| H6 | search_hash が rainbow table で逆引き可能 | security | HMAC-SHA256 + 専用 secret（SHA-256 単独ではなく）。検索ハッシュ secret の分離管理 |
| H7 | Webhook 認証が不十分 | security | LINE: X-Line-Signature 検証必須。Email: per-tenant Bearer token。dual-secret ローテーション対応 |
| H8 | 監査ログの改竄耐性なし | security | スキーマ定義を Phase 0 で確定。BigQuery sink は Phase 7 でよいが、ログ構造は先行確定 |
| H9 | カンバンの10フェーズがUX的に多すぎる | frontend | 表示上は 4-5 カラムに畳む（inquiry, hearing, visit, contract/lost）。内部モデルは10フェーズ維持、UIでグルーピング |
| H10 | SalesConsole/CRM 並行運用で RBAC バイパス可能 | security | CRM 有効テナントでは sales ロールの /sales アクセスを制限。feature flag で制御 |
| H11 | Firestore onSnapshot が Cookie 認証と非互換 | security | MVP では **SSE (Server-Sent Events)** を推奨。Firestore Security Rules 不要、認証は Cloud Run で完結 |

## 6. GTM 戦略への提言

### 設計書の前提 vs レビューチームの推奨

| 観点 | 設計書の前提 | レビューチームの推奨 |
|------|------------|-------------------|
| **ポジショニング** | "Per Heim CRM" (フル CRM) | Step 1: "AI Hearing Dashboard" (companion) → Step 2: "Per Heim CRM" |
| **初回外部リリース** | Phase 1-3 (顧客管理 + タスク + AI統合) | Phase 0+1+3 = "AI Dashboard"。既存CRM (カナリー等) と**併用** |
| **L1 テーブルステーク** | Phase 4-5 で追加 | Step 1 では不要（既存CRM側がカバー）。Step 2 で追加 |
| **差別化の訴求** | CRM 全体として | Step 1: 「Per Heim 経由の来店率が N% 向上」をデータ実証 → Step 2: CRM 統合 |

### 推奨リリースシーケンス

```
Step 1: AI Hearing Dashboard (Phase 0+1+3)     ~8-10 週
  - 既存 CRM 併用。AI プロファイル + ニーズ構造タグ + カンバン
  - 価値: 「AI が学んだ顧客理解を可視化」

Step 2: + タスク管理 (Phase 2)                  +2 週
  - 「次に何をすべきか」を管理

Step 3: + 追客 (Phase 4)                        +4-6 週
  - 通話記録、メール、LINE
  - ここから CRM 単独利用が可能に

Step 4: Full CRM (Phase 5-7)                    +8-12 週
  - 反響自動取込、分析、連携
  - ポジショニングを "Per Heim CRM" に切替
```

## 7. Go Condition への提言

### Condition 1: MVP-P1/P2 優先度分割

**推奨**:
- **P1 (内部テスト)**: Phase 0 + Phase 1 + Phase 3 = 顧客管理 + AI統合
- **P2 (初回外部リリース)**: + Phase 2 + Phase 4 (通話記録のみ) = "AI Dashboard" + タスク

### Condition 2: F3-2 API スキーマ v1

**Phase 0 で解決可能** (1-2日):
```python
class HearingCompletionEvent(BaseModel):
    """complete_session() → CRM 通知のペイロード契約"""
    tenant_id: str
    lead_id: str
    session_id: str
    customer_id: str | None = None
    hearing_profile: dict
    needs_tags: NeedsTags | None = None
    completion_rate: float  # 0.0-1.0
    completed_at: datetime
```
これを `src/server/crm/models.py` に定義し、AI ヒアリング側の output と CRM 側の input の契約とする。

### Condition 3: ユーザーストーリー

**Phase 0 で解決可能** (2-3日)。ただし「フル CRM の1日」ではなく「AI Dashboard 併用の1日」として書くべき:

**担当者の1日 (companion mode)**:
1. 朝 -- 既存 CRM で反響確認 → Per Heim Dashboard でタスク確認
2. 新規反響 → 既存 CRM で対応 → Per Heim で AI ヒアリング URL 発行
3. ヒアリング完了通知 → Dashboard で AI プロファイル確認 → 物件提案準備
4. 電話 → 既存 CRM に通話記録（Phase 4 以降は Per Heim CRM 内で完結）

**管理者の1日 (companion mode)**:
1. 朝 -- Dashboard でカンバン確認（ヒアリング進捗）
2. AI プロファイル完了率の確認
3. 月次 -- Per Heim 効果測定（Phase 6 以降）
